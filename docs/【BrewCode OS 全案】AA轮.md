# BrewCode OS 全案

## 一、创世故事

**标题：BrewCode OS 创世记——一个自由人的咖啡语言**

**序章：问题的诞生**

2026年，一个身在中国农村的自由人，面对三台电脑和正在学习的AI建站能力，问出了一个看似简单的问题：“我能否不用卖时间，而是用数字资产获得被动收入？”

这个问题在互联网上被反复提问，得到的答案千篇一律：做任务、接外包、卖课程。这些答案都有一个共同前提——你必须用时间换钱。而他想的是另一个方向：有没有一种资产，一次创造，持续产生价值？

这不是一个关于“如何赚钱”的问题。这是一个关于“如何创造”的问题。

**第一章：意外的发现**

在深度调研了图书市集、中古二手、艺术文创、手办、手帐、烘焙、咖啡、非遗、农产品、义乌小商品共十个领域后，一个残酷但清晰的结论浮现：

* 非遗和手工艺人的知识是口口相传的，没有结构化数据
* 中古和手办的鉴定数据被少数平台垄断，且90%是非结构化对话
* 咖啡和烘焙，拥有全球海量的、高度结构化的公开训练数据

这不是一个品味的选择，这是一个数据的判决。

咖啡，被选中了。

**第二章：重新定义问题**

真正改变一切的，是对问题本身的颠覆性重构。

传统思维问的是：“如何做一个咖啡内容网站，通过广告和联盟赚流量费？”

重构后的问题变成了：“咖啡是唯一将主观风味体验，完全建立在客观物理参数上的领域。如果风味可以被参数化，参数就可以被编程。那么，咖啡的终极数字形态是什么？”

答案出现了：**一个由AI驱动的、开源的、风味参数交易与复现系统。** 这不叫“咖啡App”，这叫“咖啡操作系统”。这就是BrewCode OS。

**第三章：站在巨人的肩膀上**

创造者开始寻找BrewCode OS在人类文明史上的对应者。他找到了三个：

* **GitHub：** 用版本控制统一了全球程序员的协作语言。BrewCode OS将用`.brew`参数文件统一全球咖啡爱好者的冲煮语言。
* **苹果：** 将极其复杂的Unix内核封装为触控界面。BrewCode OS将把流体力学和化学方程封装为“这杯偏苦？调粗研磨”的自然语言。
* **IEEE：** 它不生产任何电子产品，但制定了Wi-Fi协议，所有公司必须遵循。BrewCode OS不卖任何咖啡，但定义了`.brew`标准。

但最深刻的映射，来自蒂姆·伯纳斯-李。1989年，他在CERN发明了HTTP协议、HTML语言和URL地址。他没有申请专利，没有成立公司，只是把它扔向世界，说：“这是为所有人准备的。”后来，苹果、谷歌、亚马逊……所有万亿帝国都运行在他的发明之上。

BrewCode OS的创始人在笔记中写道：“我们不是要做咖啡界的GitHub。我们要做的是咖啡界的万维网。”

## 二、BrewCode OS 开发宣言

**版本：v0.1许可：Creative Commons Zero (CC0)**

### 前置声明

本文档是BrewCode OS的起源文件。它不是商业计划书，不是融资BP，不是产品需求文档。它是一份**开源咖啡运动的独立宣言**。

### 第一条：基本原理

我们相信：

1. **咖啡是物理，不是玄学。** 一杯咖啡的风味，是其研磨度、水温、粉水比、萃取时间的必然结果。任何可描述的风味差异，必然对应一组可测量的参数差异。这是“风味-参数同构原理”。
2. **冲煮知识应该是公共财产。** 没有人应该因为不知道“正确的”参数而喝不到好咖啡。世代传承的冲煮智慧，不应被封装在专利里。
3. **工具的存在是为了被遗忘。** 最好的冲煮工具，在使用时会“隐退”，让使用者直接与咖啡对话。参数界面的设计，永远服务于人的感官，而非技术的炫耀。
4. **个体拥有数据的绝对主权。** 你的冲煮记录、味觉偏好、设备参数，都属于你。任何分析、推荐、优化，都应基于你的明确授权。

### 第二条：核心定义

**BrewCode OS** 是一个致力于为全球咖啡冲煮建立通用数字语言的开放项目。

**`.brew`文件** 是这一通用语言的载体。它是一个开放的、JSON结构的、可被任何程序读写和执行的冲煮方案标准格式。

**BrewPlayer** 是`.brew`文件的官方运行时环境。

**BrewRepo** 是`.brew`文件的公共版本库。

### 第三条：社区契约

1. **开放优先：** 所有核心标准、协议、参考实现，永远以公共领域或最宽松的MIT许可发布。
2. **协作优先：** 改进从Fork开始，共识通过Pull Request达成。
3. **尊重优先：** 每个人的舌头都是有效的传感器。没有愚蠢的问题，只有傲慢的回复。
4. **隐私优先：** 用户数据采用最高标准的匿名化与加密。永远不售卖、不出租。

### 第四条：商业边界

BrewCode OS本身永远免费、开源。

商业收入仅通过以下方式获取：

* 为用户提供**可选的增值服务**（AI诊断、高级分析）
* 为硬件厂商提供**BrewCode兼容认证**和技术授权
* 发布**基于匿名聚合数据的年度报告**
* 接受社区的**自愿赞助和捐赠**

任何商业行为，不得破坏标准的开放性和数据的中立性。

### 第五条：对未来的承诺

当最后一台咖啡机停止运转，

当最后一支咖啡豆被采摘，

当所有咖啡馆都消失——

BrewCode OS所记录的每一杯咖啡的参数和背后的故事，

将成为人类感官文化史的永久见证。

我们不是在做一个App。

我们在为人类数千年的咖啡文明，

建立它的数字图书馆。

**这是BrewCode OS的誓言。**

## 三、产品架构

### 核心原则

BrewCode OS 不做“大而全”的平台，不做咖啡版的小红书，不做豆子电商。我们做的是三个东西的精确组合：一个开源标准、一套围绕标准的工具、一层跑在工具之上的AI服务。这三层互不绑定，但合在一起构成完整闭环。

### 第一层：`.brew` 标准文件

这是整个生态的原子单位。一个 JSON 文件，描述“如何冲一杯咖啡”所需的全部参数。

**结构（v0.1）：**

```json
{
  "brew": "0.1.0",
  "meta": {
    "title": "April Brewer 浅烘埃塞 经典方案",
    "author": "brewcode-os/genesis",
    "source": "https://brewcode.os/r/genesis/001",
    "license": "CC0",
    "created": "2026-06-12",
    "tags": ["light-roast", "april-brewer", "ethiopian", "floral"]
  },
  "coffee": {
    "name": "Ethiopia Yirgacheffe Washed",
    "roaster": "Apollon's Gold",
    "roast_date": "2026-05-20",
    "roast_level": "light",
    "varietal": "Heirloom",
    "process": "washed"
  },
  "equipment": {
    "brewer": "April Brewer",
    "filter": "April paper filter",
    "grinder": "Comandante C40",
    "kettle": "Fellow Stagg EKG"
  },
  "recipe": {
    "coffee_weight_g": 12,
    "water_weight_g": 200,
    "ratio": "1:16.67",
    "water_temperature_c": 93,
    "grind_setting": "Comandante 22 clicks"
  },
  "steps": [
    {"action": "rinse_filter", "note": "用热水冲洗滤纸并预热分享壶"},
    {"action": "bloom", "water_g": 30, "duration_s": 30, "note": "中心注水，确保所有粉湿润"},
    {"action": "pour", "water_g": 70, "duration_s": 15, "note": "绕圈注水至100g"},
    {"action": "pour", "water_g": 100, "duration_s": 15, "note": "中心注水至200g"},
    {"action": "drawdown", "target_s": 150, "note": "总萃取时间约2分30秒"}
  ],
  "result": {
    "tds": 1.32,
    "extraction_yield": 20.4,
    "flavor_notes": ["佛手柑", "茉莉", "柠檬酸质", "绿茶余韵"],
    "body": "轻盈",
    "acidity": "明亮",
    "sweetness": "中等"
  }
}
```

**设计决策：**

* `equipment` 用字符串而非枚举ID：v0.1先用自由文本，等数据量足够时再迁移到结构化ID。
* `steps` 是顺序数组：冲煮是时序过程，数组保证执行顺序。
* `result` 可选：方案发布时可能没有测量数据，冲完后可补充。
* 选用JSON而非YAML：JSON的生态工具链全球最完整，任何语言零依赖解析。

### 第二层：工具矩阵

**工具一：BrewPlayer（播放器）**

* **给谁用：** 只想跟着方案冲出一杯好咖啡的人。
* **做什么：** 载入`.brew`文件，渲染为分步倒计时的冲煮向导。
* **界面：** 极简。方案名、作者、当前步骤大字提示、倒计时条、下一步按钮。
* **技术：** 纯前端静态页面，PWA离线可用，部署在Vercel/Cloudflare Pages。

**工具二：BrewForge（编译器）**

* **给谁用：** 想自己设计、修改、测试冲煮方案的人。
* **做什么：** 可视化`.brew`文件创建和编辑工具。左侧编辑器，右侧实时预览。
* **关键功能：** 步骤可视化拖拽排序、参数校验、版本管理、一键导出。

**工具三：BrewRepo（仓库）**

* **给谁用：** 想发现新方案、分享自己方案的人。
* **做什么：** `.brew`文件的公共托管和协作平台，直接映射GitHub协作模型。
* **核心机制：** Repo、Fork、Pull Request、Issues——全部由GitHub原生承载。
* **搜索维度：** 按豆种、烘焙度、器具、研磨机、风味标签。

**工具之间的关系：** 三个工具各自独立，通过`.brew`文件格式互通。没有人必须三个都用。

### 第三层：AI服务

**服务一：AI风味诊断（核心付费产品）**

* **输入：** 用户自然语言问题 + `.brew`文件 + （经授权的）历史数据。
* **输出：** 具体到操作层面的调整建议。
* **价格：** v0.1按次付费，每次诊断相当于一杯好咖啡的价格。
* **隐私：** 优先本地运行；云端可选且需显式授权；永远不卖数据。

**服务二：AI方案生成（增值服务）**

* **输入：** 豆子信息 + 用户设备 + 口味偏好。
* **输出：** 可直接载入BrewPlayer的`.brew`文件初稿。
* **核心能力：** 精确到具体设备和豆子的起点方案。

## 五、服务分层设计

### 理论基底

**基底一：数据主权架构（吸收Solid项目的核心原则）**&#x6570;据与应用分离。用户的数据存在自己控制的“数据舱”里，应用只是被临时授权访问。BrewCode OS不拥有任何用户数据。

**基底二：无服务器边缘计算的服务网格**服务治理逻辑从应用代码中剥离。所有API路由通过一个网关函数分发，鉴权和限流在这里完成，业务函数只接收已认证的请求。

**基底三：关联数据的轻量化语义层**`.brew`文件可选嵌入JSON-LD语义标注，使“研磨度22格”在不同设备间可比较。这是“通用语言”愿景的技术实现。

### 分层架构

```
┌─────────────────────────────────────────────┐
│  第五层 交互与分发                           │
│  BrewPlayer · BrewForge · BrewRepo Web      │
│  纯静态资源，CDN边缘缓存，PWA离线可用        │
├─────────────────────────────────────────────┤
│  第四层 API网关（服务网格控制面）            │
│  鉴权 · 限流 · 路由 · 日志 · 版本管理        │
│  Cloudflare Worker 单文件部署               │
├─────────────────────────────────────────────┤
│  第三层 业务服务（无服务器函数）             │
│  AI诊断 · 方案生成 · 语义翻译 · 搜索 · Webhook│
│  每个服务独立函数，独立部署，独立扩缩容      │
├─────────────────────────────────────────────┤
│  第二层 数据与标准                           │
│  .brew JSON Schema · JSON-LD语义层          │
│  GitHub仓库（文件本体） · SQLite（元数据索引） │
├─────────────────────────────────────────────┤
│  第一层 基础设施                             │
│  Cloudflare Pages/Vercel · GitHub · 互联网   │
└─────────────────────────────────────────────┘
```

### 逐层设计摘要

**第五层：** 纯静态资源，CDN边缘缓存，PWA离线可用。版本化URL保证旧链接不失效。

**第四层：** 单一网关函数（\~100行），统一处理鉴权、限流、路由。API路径含版本号，旧版本保持运行12个月。

**第三层：** 五个独立无服务器函数，无状态，30秒超时，结构化JSON日志。代码量合计\~1200行。

**第二层：** JSON Schema定义标准；SQLite存元数据索引（FTS5全文搜索）；设备注册表用YAML维护。

**第一层：** Cloudflare Pages + Vercel双平台备选，GitHub承载标准版本和协作，总依赖项不到10个。

### 明确不引入的

* 不引入容器编排（无持久服务需要编排）
* 不引入独立认证服务（GitHub OAuth + API Key网关验证足够）
* 不引入微服务编排引擎（四个函数用静态路由足够）
* 不引入专属数据分析管道（初期日志量极少，手动grep足够）

## 六、商业生态文明全景

### 核心承诺：绝对不做的五件事

1. **不卖数据。** 用户数据归用户，AI在授权下读取，绝不存储出售。
2. **不卖豆子、不卖器材、不做电商。** 一旦出现购买链接，中立性崩塌。
3. **不做广告。** 广告需要注意力最大化，与“冲煮时扫一眼就够”的设计哲学冲突。
4. **不融资。** 成本极低，不需要外部资本。外部资本引入的控制权是致命的。
5. **不建封闭花园。** `.brew`永远纯文本JSON，不加密，不设DRM。

### 护城河

1. **信任护城河：** 巨头的商业模式决定了它们做不到中立。
2. **标准锁定效应：** 类似HTTP协议，替换成本随网络规模指数级上升。
3. **社区归属感：** 用户是共建者，不是消费者。商业公司买不到。
4. **成本优势：** 一个人+三台电脑+AI，大公司的组织逻辑不允许这种极致的轻资产。

## 八、风险清单与缓解策略
