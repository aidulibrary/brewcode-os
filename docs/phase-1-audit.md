> **版本：** v1.0
> **日期：** 2026-06-18

## 一、审计总览

| 层级 | 完成度 | 评级 | 一句话摘要 |
|------|--------|------|-----------|
| 标准层 | 85% | 良好 | Schema 定义完整可用，缺少英文版规范文档 |
| 工具层 | 70% | 中 | Player/Repo 交付完整，Forge 差距大，Portal 已迁移完成 |

## 二、标准层审计

### 2.1 已完成交付

| 项目 | 说明 | 状态 |
|------|------|------|
| brew.schema.json | JSON Schema Draft 2020-12，定义 6 个顶层字段（meta/coffee/equipment/recipe/steps/result），含完整字段描述和约束 | 完成 |
| CC0 许可证 | Schema 以 CC0 1.0 发布，公共领域 | 完成 |
| 种子方案校验 | 50 个种子文件全部通过 Schema 校验 | 完成 |
| Forge 内置校验 | forge.js 中集成了 Schema 实时校验逻辑 | 完成 |

**关键资产**：[brew.schema.json](file:///d:/brewcode-os/packages/standards/brew.schema.json) — 约 400 行，定义 13 种步骤操作类型（prepare/rinse/grind/bloom/pour/stir/swirl/drawdown 等），覆盖手工冲煮全部场景。

### 2.2 未闭合项

| 项目 | 当前状态 | 遗留问题 |
|------|---------|---------|
| 英文版 Schema 规范文档 | 未开始 | Schema 注释为中文，国际开发者理解成本高。缺少 RFC 风格的独立规范文档（类似 JSON Schema 的 spec.md） |
| Schema 版本号策略 | brewCodeVersion 为 "0.1"，但无版本演进规则 | 缺少 SEMVER 策略说明——什么算 breaking change、什么算 minor |
| $id 域名 | 指向 `brewcode.dev`，该域名不属于项目 | 应改为 `schemas.brewcode.礼字号.中国` 或 `brewcode.礼字号.中国/schemas/brew.schema.json` |

### 2.3 技术债务

| 问题 | 严重度 | 说明 |
|------|--------|------|
| Schema 缺少 examples 字段 | 低 | JSON Schema 支持 `examples` 字段，可内嵌示例值，降低学习成本 |
| result 字段为可选 | 低 | 当前 `result` 不在 `required` 中，但种子文件中 result 字段一致性差——部分文件有、部分无 |
| 缺少格式校验 | 中 | 如 `roastDate` 仅定义为 string，未约束为 date 格式。`createdAt` 同理 |

### 2.4 优先级建议

| 优先级 | 动作 | 理由 |
|--------|------|------|
| 高 | 撰写英文版 Schema 规范文档 | 标准走向国际化的基础设施，不可跳过 |
| 中 | 修正 `$id` 域名 | 避免依赖不属于项目的域名 |
| 低 | 补充 examples 和格式校验 | 降低集成门槛，但非阻断 |

### 3.3 BrewForge（方案编译器）

> **重要提示**：以下评估基于 2026-06-15 的验证报告（[forge-validation-report.md](file:///d:/brewcode-os/docs/forge-validation-report.md)），该报告指出 Forge 当前实现与 MVP 目标之间存在巨大差距。

#### 已完成交付

| 功能 | 说明 | 状态 |
|------|------|------|
| 基础表单 UI | 4 个分区（方案信息/咖啡豆/器具/冲煮参数），41 个输入控件 | 完成 |
| DOM ↔ State 同步 | 表单输入实时同步到 editorState 对象 | 完成 |
| 暗色主题 | 基础 CSS 暗色主题 | 完成 |
| 移动端适配 | `@media (max-width: 640px)` 单列布局 | 完成 |
| 域名上线 | forge.礼字号.中国 正常访问 | 完成 |

#### 未闭合项（4 大核心模块全部缺失）

| 模块 | 缺失内容 | 验证报告结论 |
|------|---------|-------------|
| 代码模式 | 无 CodeMirror 集成、无模式切换按钮、无 JSON 代码编辑区 | 3.1–3.6 全部未实现 |
| 步骤管理器 | 无步骤列表、无添加/编辑/删除步骤、无模态框 | 4.1–4.8 全部未实现 |
| 实时校验 | 无 validate 函数、无状态栏、无错误计数、无红框提示 | 5.1–5.7 全部未实现 |
| 导入/导出 | 无导出按钮、无 .brew 文件下载、无从 Player 跳转 | 10.1–10.4 全部未实现 |

**验证报告评分**：功能性 2/10，稳定性 7/10，用户体验 3/10。8 项通过，25 项失败，4 项建议改进。`forge.js` 仅 219 行，核心功能代码量为零。

#### 技术债务

| 问题 | 严重度 | 说明 |
|------|--------|------|
| 无 localStorage 持久化 | 高 | 刷新页面所有输入丢失，用户体验极差 |
| editorState 初始无步骤 | 中 | `steps` 初始化为 `[]`，即使后续实现步骤管理器也应至少有一个空步骤模板 |
| createdAt 固定为页面加载时刻 | 低 | 刷新后时间戳变化，若实现 localStorage 恢复后 createdAt 应保持不变 |
| Footer 文案误导 | 低 | "所有输入即时保存到状态中"——用户理解为已持久化，但实际仅内存同步 |

#### 优先级建议

| 优先级 | 动作 | 理由 |
|--------|------|------|
| 高 | 实现步骤管理器（添加/编辑/删除步骤） | Forge 核心价值是"创建 .brew 方案"，没有步骤管理器等于不能创建方案 |
| 高 | 实现导出功能（下载 .brew 文件） | 创建完方案后必须能导出，否则整个工具无意义 |
| 高 | 添加 localStorage 持久化 | 防止刷新丢失数据，最低成本提升体验 |
| 中 | 集成 CodeMirror 代码模式 | 双模式编辑是 Forge 的差异化特性 |
| 中 | 实现实时 Schema 校验 | 保证输出数据质量，区分"能用"和"好用" |
| 低 | 实现 AI 功能集成 | 依赖 AI 服务层恢复，非 Forge 自身问题 |

### 4.2 未闭合项

| 项目 | 当前状态 | 遗留问题 |
|------|---------|---------|
| DeepSeek API Key | 已删除/失效 | `/diagnose` 和 `/generate` 端点返回 502。需重新申请 Key 并更新 Worker 环境变量 |
| 鉴权密钥轮换 | 测试密钥 `bk_test1234567890abcdef` 未替换 | 生产环境仍使用开发测试密钥 |
| 无余额监控 | 无 | 上次 Key 失效可能因余额不足未及时发现 |

### 4.3 技术债务

| 问题 | 严重度 | 说明 |
|------|--------|------|
| wrangler.toml 硬编码 API Key | 高 | `DEEPSEEK_API_KEY = "REDACTED"` 直接写入配置文件，已提交 Git。违反项目自身安全原则（domain-infra-ops.md 明确写"绝不写入 Git 仓库"） |
| 鉴权哈希算法过于简单 | 高 | 使用 32-bit 整数哈希，不抗碰撞攻击。生产环境需升级为 SHA-256 + salt |
| 诊断 Worker 无缓存 | 中 | 相同方案+相同问题可能重复调用 DeepSeek，浪费 API 额度 |
| 生成 Worker 无 Schema 版本校验 | 中 | 若 Schema 升级，生成 Worker 的 BREW_SCHEMA_TIPS 可能不同步 |
| 翻译 Worker 无模糊匹配 | 低 | 用户输入的设备名称需精确匹配，不支持模糊搜索 |

### 4.4 优先级建议

| 优先级 | 动作 | 理由 |
|--------|------|------|
| 高 | 申请新 DeepSeek API Key，通过 Cloudflare Dashboard 环境变量注入（不写入 wrangler.toml） | 恢复 AI 服务层核心功能，同时修复安全漏洞 |
| 高 | 升级鉴权哈希算法为 SHA-256 | 生产环境安全基线 |
| 中 | 生成生产环境 API Key 替换测试密钥 | 防止测试密钥泄露导致的滥用 |
| 中 | 添加余额监控（Worker 内调用 /user/balance 或在 DeepSeek 平台设置告警） | 避免下次余额耗尽才发现 |
| 低 | 翻译 Worker 添加模糊匹配 | 提升用户体验 |

| 静态资源无版本号 | 中 | Player/Repo 的 CSS/JS 已加 `?v=` 参数，但 Forge 和 Portal 未加，修改后可能因 CDN 缓存不更新 |
| wrangler.toml 中 API Key 硬编码 | 高 | 见 AI 服务层技术债务 |

### 6.4 优先级建议

| 优先级 | 动作 | 理由 |
|--------|------|------|
| 高 | 从 wrangler.toml 移除 API Key，改用 Cloudflare Dashboard 环境变量 | 安全基线，不可拖延 |
| 中 | 为 Forge/Portal 静态资源添加版本号 | 避免部署后用户看到旧版 |
| 中 | 接入 UptimeRobot 或 Cloudflare 自带监控 | 低成本获得站点可用性感知 |
| 低 | 制定 D1 备份策略 | 当前数据量小，可暂缓 |
