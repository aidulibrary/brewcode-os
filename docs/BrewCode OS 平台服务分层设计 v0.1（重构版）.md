## BrewCode OS 平台服务分层设计 v0.1

### 设计前置：为什么分层

一个咖啡参数平台不需要像 AWS 那样的全球基础设施分层。但 BrewCode OS 有一个特殊挑战：它必须同时服务四种行为模式截然不同的用户。

* **冲煮者**：打开 BrewPlayer，跟着步骤走。需要的是零延迟、离线可用。
* **创作者**：在 BrewForge 里编辑方案。需要的是实时校验、版本对比、一键发布。
* **开发者**：调用 `.brew` 标准做自己的工具。需要的是稳定的 API、清晰的文档、不破坏的向后兼容。
* **AI 服务消费者**：提交诊断请求。需要的是低延迟、隐私保护、结果可解释。

如果架构不分层，这四个场景的实现代码会互相缠绕。改 Player 的 UI 可能影响诊断 API 的响应格式。这种耦合在一个人维护的项目里是致命的。

分层不是为了显得专业，是为了让每一层可以独立演进、独立部署、独立失败而不波及其他。

### 理论基底：三则被验证但未被执行的前沿思想

以下三个理论框架已经在产业界被充分讨论和局部验证，但由于组织惯性和遗留系统负担，几乎没有公司真正完整实施过。BrewCode OS 从零开始，没有历史包袱，可以首次将它们完整落地。

#### 基底一：SOLID 原则下的数据主权架构

* **来源**：Tim Berners-Lee 在 MIT 发起的 Solid 项目及其身后的语义网哲学。
* **核心主张**：数据与应用分离。用户的数据存在自己控制的“数据舱”（Pod）里，应用只是被临时授权访问。用户随时可以撤销权限，带走数据。
* **为什么产业界没执行**：现有商业模式的根基是“收集用户数据-分析-变现”。放弃数据控制权等于放弃核心资产。没有公司愿意主动做这件事。
* **为什么 BrewCode OS 可以**：我们不靠卖数据赚钱。我们的收入来自增值服务（AI 诊断）和认证费。用户的数据存储在 GitHub 的公共仓库（显式选择开源）或用户自己的存储中（私有方案）。BrewCode OS 本身不拥有任何冲煮数据。从架构层面，Solid 的“Pod 概念”可以直接复用：BrewRepo 上的每一个用户或组织，就是一个数据舱。`.brew` 文件是舱内的资源。应用的访问通过标准 API（GitHub API）而非私有数据库查询。
* **落地方式**：不完全实现 Solid 协议（太重），但吸收其核心原则——“数据与应用分离”作为整个分层架构的第一约束。

#### 基底二：无服务器边缘计算的服务网格

* **来源**：Google 的 ServiceWeaver、Cloudflare Workers 生态、以及 eBPF 驱动的零信任网络思想。
* **核心主张**：服务治理逻辑（路由、鉴权、限流、可观测性）从应用代码中剥离，下沉到基础设施层。对于小团队，这意味着每个无服务器函数只管自己的业务逻辑，网格层自动处理 HTTPS、CORS、API Key 校验、调用链追踪。
* **为什么产业界没完整执行**：需要统一的控制面和数据面。大公司内部的服务网格通常只覆盖集群内的服务，边缘节点的网格化还在探索期。
* **为什么 BrewCode OS 可以**：我们只有一类运行环境——无服务器函数（Vercel Functions / Cloudflare Workers）。Cloudflare Workers 本身内置了基于 Anycast 的全球路由网络，每个请求自动到达最近节点。我们不需要自己搭建服务网格，只需要利用平台的网格能力。具体来说：所有 API 路由统一通过一个网关函数分发，鉴权和限流在这里完成，业务函数只接收已认证的请求。
* **落地方式**：API 网关函数 + 后端业务函数，构成一个极简的服务网格。网关函数不到100行，复用 Cloudflare 的边缘网络能力。

#### 基底三：关联数据与语义网的轻量化落地

* **来源**：万维网联盟的 RDF、JSON-LD、Schema.org 标准。
* **核心主张**：数据不只是给人看的文档，也应该能被机器理解其含义。一杯咖啡的“烘焙日期”不是任意字符串，而是一个类型为 `schema:productionDate` 的日期值。当数据携带语义后，不同平台的数据可以自动互操作。
* **为什么产业界只部分执行**：SEO 驱动的网站大量使用 JSON-LD 做结构化标记（为了 Google 富文本摘要），但仅此而已。没有人真正用它来做跨平台的数据互操作，因为商业动机是“把数据关在自己的围墙里”。
* **为什么 BrewCode OS 可以**：`.brew` 文件天然携带语义需求。“研磨度 22 格”在不同磨豆机之间无法直接比较。如果我们在标准中引入语义标注——22 clicks on Comandante C40，颗粒度约 800 微米——那么 AI 就可以在不同设备之间做翻译。这种语义层是 BrewCode OS 的“通用语言”愿景的技术实现。
* **落地方式**：`.brew` 标准中可选嵌入 JSON-LD 语义标注字段，v0.2 引入，v0.1 预留字段名空间。不引入完整的 RDF 数据库，太重的存储和查询开销。语义映射表由一个社区维护的 YAML 文件定义，存储在 GitHub，随版本演进。

### 逐层设计

#### 第五层：交互与分发

**职责**：用户看到和触摸到的一切。

**设计决策**：

* **纯静态资源**。BrewPlayer 是一个 HTML 文件。BrewForge 是一个 HTML 文件。没有服务端渲染，没有模板引擎运行时。
* **CDN 边缘缓存**。所有静态资源通过 Cloudflare 全球 CDN 分发。一个中国用户和一个巴西用户访问 BrewPlayer，都从最近的节点加载，延迟低于 50ms。
* **PWA 离线优先**。Service Worker 缓存关键资源。BrewPlayer 在没有网络的场景下依然可用——冲煮咖啡的地方可能是地下室、偏远产区、户外市集。离线体验是核心需求，不是锦上添花。
* **版本化的资源 URL**。`/player/v0.1.0/` 和 `/player/v0.2.0/` 是不同的路径。旧版本永远不被删除。这样嵌入旧版 `.brew` 文件的链接不会因为升级而失效。

**耦合关系**：第五层只依赖第二层的数据（通过第四层的 API 获取），不直接依赖第三层的任何服务。播放器不需要知道 AI 诊断是否存在。

#### 第四层：API 网关

**职责**：所有后端服务的统一入口。这是 BrewCode OS 架构中唯一需要运行在服务器端的常驻组件。

**为什么需要这一层**：

* v0.1 有至少四个后端服务（AI 诊断、方案生成、索引搜索、Webhook 处理）。如果每个服务独立暴露 URL，前端需要知道四个地址、管理四套认证。
* 当服务需要拆分或合并时，前端代码必须更新。这对于一个鼓励第三方开发者接入的生态而言是不可接受的。
* 网关层把“服务的物理部署”和“服务的逻辑接口”解耦。

**设计决策**：

* **单一网关函数**。一个 Cloudflare Worker 脚本，接收所有 `/api/*` 的请求，解析路径，转发到对应的后端函数，返回响应。这个脚本预计 100 行。
* **认证在这里统一处理**。API Key 放在请求头 `Authorization: Bearer bk_xxx` 中，网关验证后把用户标识注入请求上下文。后端函数收到的请求已经经过认证，不需要重复验证。
* **限流也在这里**。用 Cloudflare KV 做简单的计数器，每个 API Key 每分钟最多 60 次调用。超过限制返回 429。这是一个“够用”的方案，不需要引入专门的限流服务。
* **版本管理**。API 路径包含版本号：`/api/v0/`。当 API 需要不兼容升级时，部署 `/api/v1/`，旧版本保持运行 12 个月后退役。版本退役的公告通过 GitHub Discussions 发布，也写入 API 响应头 `X-API-Deprecated: 2027-06-13`。

**为什么不使用 API 网关产品（如 Kong、AWS API Gateway）**：

* 对于一个人维护的项目，它们太重。Kong 需要自己的数据库和配置管理。AWS API Gateway 把项目锁定在 AWS 生态。
* 100 行的 Cloudflare Worker 可以完成同样的事，且部署在边缘网络上，全球延迟一致低。

#### 第三层：业务服务

**职责**：执行具体的业务逻辑。每个服务是一个独立的无服务器函数。

**服务列表（v0.1）**：

| **服务名**              | **触发方式**       | **职责**                                  | **预估代码量** |
| -------------------- | -------------- | --------------------------------------- | --------- |
| `ai-diagnose`        | HTTP POST      | 接收 `.brew` 文件 + 用户描述，返回诊断建议             | \~300行    |
| `recipe-generate`    | HTTP POST      | 接收豆子信息 + 设备信息，生成初始方案                    | \~300行    |
| `semantic-translate` | HTTP GET       | 将设备名称翻译为跨品牌的可比较参数（如“C40 22格 ≈ 粒径800μm”） | \~200行    |
| `index-search`       | HTTP GET       | 搜索已索引的方案元数据                             | \~100行    |
| `webhook-handler`    | GitHub Webhook | 响应 Repo 变更事件，触发索引重建                     | \~80行     |

**设计决策**：

* **每个服务独立部署**。修改 AI 诊断的提示词不会影响搜索索引的重建。
* **无状态**。所有状态存储在第二层（GitHub、SQLite、KV）。服务函数本身可以随时被销毁和重建。
* **超时控制**。AI 调用的延迟不确定性来自外部模型 API。每个服务设置 30 秒超时，超时返回 504，由前端决定是否重试。重试逻辑在网关层统一实现（指数退避，最多3次）。
* **可观测性**。每个函数的日志输出结构化的 JSON 行，包含 `request_id`、`service_name`、`duration_ms`、`status`。用 Cloudflare 的日志存储，不引入专门的监控系统。v0.1 阶段，“能看到日志”就够了，不需要仪表盘。

#### 第二层：数据与标准

**职责**：定义什么是合法数据，存储数据本体，提供快速检索。

这是 BrewCode OS 最核心的一层。代码会重写，服务会替换，但数据标准是生态的地基。

**子层 2A：标准定义**

* `.brew` JSON Schema：数据格式的宪法。定义哪些字段是必需的，哪些可选，值的类型和范围。
* JSON-LD 上下文文件：为 `.brew` 文件中的字段添加语义标注。`"grind_setting"` 在 JSON-LD 中被映射为 `brewcode:grindSetting`，并关联到设备注册表中的具体型号。
* 设备注册表：一个由社区维护的 YAML 文件，记录每个设备的可比较参数。Comandante C40 的“1格” ≈ 30微米粒径变化。Fellow Ode 的“1档” ≈ 50微米。这个注册表是 AI 做跨设备推荐的基础。
* **版本控制**：所有标准文件存储在 `github.com/brewcode-os/standards` 仓库中，使用语义版本号（SemVer）。标准的修改必须通过 Pull Request，附有实例说明和迁移计划。这对保证生态稳定性至关重要。

**子层 2B：数据存储**

* `.brew` 文件本体：存储在 GitHub 仓库中（公共方案在 `brewcode-os/brewrepo`，用户私有方案在自己的仓库中）。
* 元数据索引：一个 SQLite 数据库，从所有公共 `.brew` 文件中提取可搜索字段（标题、作者、豆子、器具、风味标签），通过 GitHub Action 定期重建。
* **为什么是 SQLite 而不是 Elasticsearch**：1万条方案的数据量，SQLite 的全文搜索（FTS5 扩展）已经足够快且足够精准。Elasticsearch 需要 Java 运行时和至少 2GB 内存，对一个人维护的项目是灾难。当数据量增长到需要 Elasticsearch 时（百万级方案），BrewCode OS 已经成功到需要专门的团队了。

**子层 2C：缓存**

* 已索引的元数据定期导出为静态 JSON 文件，放在 CDN 上。前端搜索先请求这个静态文件（命中率高，延迟低），未命中时回退到搜索 API。这个设计把搜索的负担从无服务器函数转移到 CDN 的免费带宽上。

#### 第一层：基础设施

**职责**：为上面四层提供计算、存储、网络。

**复用清单**：

* **Cloudflare Pages**：静态资源托管 + 边缘函数。
* **Vercel**：备用部署平台，保证不被单一厂商锁定。DNS 层面可以随时切换。
* **GitHub**：代码托管、标准版本控制、协作引擎、Webhook 事件源、CI/CD。
* **互联网根基础设施**：域名（Namecheap）、DNS（Cloudflare DNS，免费且全球最快）。

**为什么只有这些**：BrewCode OS 的服务分层架构的总依赖项不到 10 个。如果 Cloudflare 明天消失，迁移到 Vercel + Netlify 需要的工作量不超过一个下午。这种“反脆弱性”不是偶然的，是刻意设计的结果。

### 分层之间如何协作：一次 AI 诊断请求的完整路径

这是一个端到端的例子，展示各层如何配合。

1. 用户在 BrewPlayer 中点击“诊断这杯咖啡”。
2. **第五层**：BrewPlayer 前端收集当前加载的 `.brew` 文件内容和用户在文本框中的描述，组装为 JSON，发送 POST 请求到 `/api/v0/ai-diagnose`。
3. **第四层**：API 网关接收请求，检查 API Key 和限流状态，将请求路由到 `ai-diagnose` 函数。
4. **第三层**：`ai-diagnose` 函数解析请求体，调用 `semantic-translate` 将设备参数标准化，然后构建包含提示词、`.brew` 文件内容和用户描述的上下文，调用 AI 模型。
5. **第二层**：在构建提示词的过程中，函数可能查询语义层的设备注册表（将“C40 22格”转换为“中等偏细研磨，约800μm”），也查询知识库中相似方案的历史诊断记录。
6. AI 模型返回结果。`ai-diagnose` 将结果格式化后返回给网关，网关返回给前端。
7. **第五层**：BrewPlayer 渲染诊断结果。

整个路径涉及的计费计算时间不超过 2 秒，涉及的服务只有 3 个（网关、诊断、语义翻译），每层的失败都可以独立处理和重试。

### 服务分层的灵魂：不做什么

架构设计的一半工作是决定做什么，另一半是决定不做什么。

**不引入容器编排**：项目不运行需要持久化的常驻服务。无服务器函数的冷启动延迟（\<100ms）在 AI 调用的上下文（总耗时 1-3 秒）中不可感知。

**不引入独立的认证服务**：BrewCode OS 的认证需求很简单：GitHub OAuth 用于 BrewRepo 的协作功能；API Key 用于后端 API。GitHub OAuth 由社区现有的库处理；API Key 验证在网关函数中用一个 30 行的 KV 查询实现。引入 Auth0 或 Supabase Auth 只会增加供应商依赖和月度账单。

**不引入微服务编排引擎**：四个无服务器函数的依赖关系在 API 网关中用静态路由表表达就足够了。不需要 Temporal、Camunda 或 AWS Step Functions。如果未来需要复杂的多步骤事务（如“生成方案→发布到 Repo→通知订阅者”），引入一个简单的 Webhook 链即可。

**不引入专属的数据分析管道**：v0.1 的日志量极少。Cloudflare 的内置日志存储 + 手动 grep 可以满足全部排障需求。当社区规模达到“需要分析用户行为以改进产品”时，再考虑引入一个轻量分析工具（如 Plausible，开源，自部署选项）。
