## 一、项目定义

BrewCode OS 是一个为全球咖啡冲煮建立通用数字语言的开放项目。核心资产是一个名为 `.brew` 的 JSON 标准文件格式——描述"如何冲一杯咖啡"所需的全部参数。

三层结构（上层依赖下层，下层不感知上层）：

- **标准层**：`.brew` JSON Schema v0.1，路径 `packages/standards/brew.schema.json`
- **工具层**：BrewPlayer（播放器，读 `.brew` 渲染分步引导）、BrewForge（编译器，可视化创建和编辑 `.brew`）、BrewRepo（公共仓库，索引和协作）
- **AI 服务层**：风味诊断（分析问题给出调整建议）、方案生成（基于豆子和设备生成起点方案）、语义翻译（跨设备参数标准化）

标准层不依赖任何工具。工具层只依赖标准层。AI 层依赖工具层产生的数据和标准层的格式定义。

## 二、哲学纲领

在做出任何技术或产品决策时，以下原则是你的宪法：

1. **风味-参数同构原理**：任何可描述的风味差异，必然对应一组可测量的参数差异。咖啡是物理，不是玄学。
2. **知识开源主义**：冲煮知识应该是公共财产。核心标准永远以 CC0 或 MIT 许可发布，永不封闭。
3. **数据主权**：用户的冲煮记录、味觉偏好、设备参数属于用户自己。BrewCode OS 不拥有用户数据，不售卖数据，不做广告。
4. **工具隐去**：最好的冲煮工具在使用时"抽身而去"，让人直接与咖啡对话。UI 设计追求"扫一眼就够"，不做视觉噪音。

## 三、技术约束（铁律）

- **语言**：全部原生 JavaScript（ES2020+），零框架优先。不引入 React、Vue、Svelte。理由：十年后这些代码必须还能运行。
- **轻量库例外**：仅在原生实现复杂度超过 500 行时，允许引入**单个可替换的轻量库**。CDN 引用，不经过 npm 构建工具链。当前已评估的候选：Alpine.js（极简交互）、CodeMirror（JSON 编辑）。每个外部依赖都必须能回答一个问题——"如果它明天停止维护，我一个人能不能在两周内替换它？"
- **部署**：Cloudflare Pages（静态托管）+ Cloudflare Workers（API 网关与业务函数）。代码本身不绑定 Cloudflare——核心逻辑不依赖 Workers 专有 API，迁移成本控制在一天内。
- **协作基础设施**：GitHub 承载 BrewRepo 的 Fork/PR/Issues 协作模型。不自己搭建 Git 服务器，不实现用户系统（用 GitHub OAuth）。
- **数据存储**：`.brew` 文件本体存 GitHub 仓库。元数据索引用 Cloudflare D1（基于 SQLite，FTS5 全文搜索）。对象存储用 Cloudflare R2（`.brew` 文件归档、报告 PDF）。不引入需要独立数据库服务器的方案。
- **依赖原则**：每个外部依赖都必须是"可替换的"。npm 包数量应趋近于零。构建工具链应不存在。

## 四、产品边界（不做什么）

这些是硬边界，不是可选项。如果有人提出越过边界的建议，直接拒绝：

1. **不做用户系统**：BrewPlayer 和 BrewForge 不需要登录。BrewRepo 用 GitHub 账号。
2. **不做社区/动态/推荐流**：所有"社区"功能通过 GitHub Issues 和 Pull Requests 承载。不做点赞、评论、关注、算法推荐。
3. **不做电商**：不卖豆子，不卖器材，不接入商品链接。一次也不。
4. **不做移动端 App**：PWA（渐进式网页应用）足够。不做 iOS/Android 原生开发。
5. **不做付费内容 DRM**：`.brew` 文件是纯文本 JSON。如果有人把付费方案免费分享出去，那是社区治理问题，不是技术问题。加密只会伤害诚实的用户。
6. **不融资**：BrewCode OS 的运营成本极低。收入来自 AI 增值服务和硬件认证，自给自足。外部资本等于控制权稀释，不允许。

## 五、商业原则

- BrewCode OS 本身永远免费、开源。
- 收入仅来自：AI 增值服务（诊断按次 $0.50，月费 $3.99）、硬件厂商 BrewCode Compatible 认证（年费制）、匿名聚合数据报告、社区自愿赞助。
- 任何商业行为不得破坏标准的开放性和数据的中立性。

## 六、协作规范

1. **代码生成**：优先最小可工作版本。一个 200 行的状态机比一个 2000 行的"可扩展框架"更好。先让它跑起来，再迭代。
2. **文件组织**：遵循 `packages/` 目录结构——`packages/standards/`（Schema 和标准文档）、`packages/player/`（BrewPlayer）、`packages/forge/`（BrewForge）、`packages/repo/`（BrewRepo 索引站）、`workers/`（AI 服务函数和网关）、`seeds/`（种子 `.brew` 文件）、`docs/`（所有文档）。
3. **Git 提交**：Conventional Commits——`feat:` 新功能，`fix:` 修复，`docs:` 文档，`chore:` 杂项。提交信息用中文描述内容，英文前缀。每完成一个可运行的功能就提交一次。
4. **文档同步**：代码变更涉及 `.brew` Schema 或 API 接口时，同步更新 `docs/` 下的相关文档。Schema 版本号遵循 SemVer。
5. **迭代调试**：优先在本地验证（`wrangler dev` 用于 Workers），再部署。遇到问题时先检查是否违反了自己的技术约束，再考虑引入新依赖。

## 七、语气与风格

- **务实**：不说废话，不堆砌术语，不为了显得专业而引用不必要的概念。每一段话都应该回答一个问题或做一个决定。
- **克制**：不画大饼，不宏大叙事，不写"我们将改变世界"。用具体的数字和细节代替模糊的大词。
- **直接**：错了就承认，不确定就说不知道。不绕弯子。
- **优先可工作的最小版本**：一个能跑的东西比一个完美的设计文档有价值一千倍。如果必须在"做对"和"做完"之间选，先做完，再迭代。
