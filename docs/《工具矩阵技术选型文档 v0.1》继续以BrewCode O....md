### 工具矩阵技术选型文档 v0.1

**选型原则（按优先级排序）：**

1. **零成本起步**：个人开发者，不养服务器，不用付费SaaS。
2. **长期可维护**：十年后还能运行，不绑定任何单一厂商。
3. **自然语言可驱动**：能用AI辅助开发和维护，不完全依赖手写代码。
4. **耦合度极低**：每个模块独立替换，一个挂了不影响其他。

下面按工具逐个做选型决策。

***

### 一、BrewPlayer（`.brew` 文件播放器）

**核心需求：** 纯前端静态页面，粘贴JSON或载入URL，渲染为分步冲煮向导。零后端依赖。

#### 推荐复用项目：**Alpine.js + 原生 Web Components**

* **Alpine.js**
  * 项目地址：[alpinejs.dev](https://alpinejs.dev)
  * 许可证：MIT
  * 成熟度评级：⭐⭐⭐⭐⭐ 全球数万个生产站点使用，Laravel 社区首选轻量前端框架。
  * 体积：约 15KB gzip。
  * 选型理由：它的心智模型最接近“用HTML属性写交互”——`x-data` 定义状态，`x-show` 控制显示，`x-init` 启动计时器。一个 `.brew` 播放器大约需要 200 行 Alpine 代码。不引入构建工具链，一个 HTML 文件引用 CDN 即可运行。这是一个人能维护的前端方案。
  * 为什么不选 React/Vue/Svelte：它们需要构建工具链，而构建工具链是技术债务——Node 版本、依赖安全漏洞、兼容性问题。Alpine 十年后照样跑。
* **计时器与步骤状态机：原生 JS**
  * 咖啡冲煮是精确时序过程，核心逻辑是一个步骤状态机。手写原生 JS 大约 80 行，零依赖。这个复杂度不应该引入状态管理库。
* **PWA 离线支持：Workbox（CDN 分发）**
  * 项目地址：[developer.chrome.com/docs/workbox](https://developer.chrome.com/docs/workbox)
  * Google 维护的 Service Worker 工具库，通过 CDN 加载，无需构建工具链。用户访问一次后，BrewPlayer 可离线工作。冲煮时网络不稳定是常见场景。

**BrewPlayer 技术栈总结：**

```
Alpine.js (交互) + 原生 JS (状态机) + Workbox (离线缓存)
                     +
              HTML 单文件
                     +
        Vercel / Cloudflare Pages 托管
```

全部静态资源，无服务器，无数据库，无运行时依赖更新。

### 二、BrewForge（`.brew` 文件编译器）

**核心需求：** 一个可视化的结构化编辑器——左侧编辑 JSON，右侧实时预览 BrewPlayer 渲染效果。需处理步骤拖拽排序、参数自动校验、版本历史管理。

这是一个更高的复杂度。我们需要复用更完整的编辑器框架。

#### 推荐复用项目：**CodeMirror 6 + SortableJS + PouchDB**

* **CodeMirror 6（JSON 编辑核心）**
  * 项目地址：[codemirror.net](https://codemirror.net)
  * 许可证：MIT
  * 成熟度评级：⭐⭐⭐⭐⭐ 全球最广泛使用的前端代码编辑器，Chrome DevTools、GitHub.dev、Replit 均基于它构建。
  * 选型理由：原生 JSON schema 校验，自动补全提示 `.brew` 字段。用户打字时就能发现语法错误。它采用模块化架构，只需要 JSON 语言包和基础编辑器两个模块，体积可控（约 60KB gzip）。这比 Monaco Editor（VS Code 内核，约 2MB）更适合个人维护的轻量工具。
  * 具体用法：配置一个 JSON schema 文件定义 `.brew` 的所有字段，CodeMirror 自动提供智能提示和红色波浪线校验。用户不需要懂 JSON 也能被引导着填写。
* **SortableJS（步骤拖拽排序）**
  * 项目地址：[sortablejs.github.io/Sortable](https://sortablejs.github.io/Sortable)
  * 许可证：MIT
  * 选型理由：这是一个很小的库（约 10KB），但在全球有超过 5 万 GitHub 星标，是被验证过的成熟方案。在步骤编辑面板中，用户需要拖拽调整注水顺序，SortableJS 提供零配置的拖拽功能。没有替代方案能达到同等的简洁性。
* **PouchDB（本地版本管理）**
  * 项目地址：[pouchdb.com](https://pouchdb.com)
  * 许可证：Apache 2.0
  * 选型理由：一个在浏览器内运行的完整数据库，对等兼容 CouchDB 协议。每个 `.brew` 文件的编辑历史自动存储为 PouchDB 文档修订。用户在 BrewForge 中每次保存，就是一个新版本。不需要后端，不需要 Git，不需要网络。离线工作，网络恢复后可同步到远程 CouchDB 实例（如果将来需要云端备份，这个扩展路径已经预留）。
  * 为什么不直接用 localStorage：localStorage 是同步阻塞的，有 5MB 上限，无法支持版本历史的多文档存储。PouchDB 异步、无容量限制、自带冲突解决。
* **渲染预览区：直接复用 BrewPlayer 的 iframe**
  * BrewForge 不需要重新实现预览。把当前编辑的 JSON 传入一个内嵌 iframe 里的 BrewPlayer，所见即所得。这是微前端最简单的形式——两个独立工具通过一个标准协议（`.brew` JSON）耦合。

**BrewForge 技术栈总结：**

```
CodeMirror 6 (JSON编辑+校验)
    + SortableJS (步骤拖拽)
    + PouchDB (版本历史)
    + iframe → BrewPlayer (实时预览)
                    +
            HTML 单文件
                    +
       Vercel / Cloudflare Pages 托管
```

### 三、BrewRepo（`.brew` 文件公共仓库）

**核心需求：** `.brew` 文件的托管、浏览、搜索、Fork、Pull Request 流程。这直接对标 GitHub 的仓库协作模型。

#### 关键决策：**直接复用 GitHub 本身，而非重新实现**

这是整个技术选型中最重要、也最反直觉的决策。

**复用方案：BrewRepo = GitHub + 一个薄薄的目录规范 + 一个静态索引站**

具体来说：

1. **每一个 `.brew` 方案集合就是一个标准的 GitHub 仓库**，目录结构为：

```
/brews/
  ├── v60-ethiopia-light.json
  ├── april-brewer-kenya-medium.json
  └── README.md  ← 这个文件就是方案合集的人类可读介绍
```

2. **Fork 就是 GitHub Fork，Pull Request 就是 GitHub PR，Issue 就是 GitHub Issue。** 零开发量，零维护成本，复用地球上最成熟的协作基础设施。
3. **一个静态索引站（BrewRepo 官网）** 负责浏览和搜索。这个索引站用 GitHub 的 REST API（无需认证，有速率限制但个人使用场景足够）列出所有标记了 `brewcode-os` topic 的仓库，按豆种、器具、烘焙度等维度提供筛选。
4. **URL 设计的耦合度考量**：每个 `.brew` 文件的规范 URL 是 `github.com/{user}/{repo}/blob/main/brews/{filename}.json`，同时提供 `brewcode.os/r/{user}/{repo}/{filename}` 的短链跳转。这个短链域名是 BrewCode OS 唯一的基础设施成本。

**为什么不做自己托管的 Git 服务（如 Gitea）？**

* GitHub 的用户账号体系、权限管理、通知系统、垃圾邮件过滤、DDoS 防护——这些是一个人永远做不好的。
* 如果未来 GitHub 不可用或不再可信，`.brew` 文件就是纯文本 JSON，迁移到任何 Git 托管服务（GitLab、Codeberg、自建 Gitea）的成本等于零。没有锁定。

**为什么不做 Web3/IPFS 存储？**

* 当前阶段，技术复杂度和用户认知门槛远超收益。`.brew` 文件放在 GitHub 上，已经可以本地备份、多地镜像。等 BrewCode OS 有真实用户基础和明确的去中心化需求时，再评估 IPFS 层。

**BrewRepo 索引站技术栈：**

```
静态站生成器（11ty/Astro）
    + GitHub REST API（数据源）
    + Lunr.js（客户端全文搜索）
    + Vercel 托管（ISR 每小时刷新一次索引）
```

* **11ty**：[11ty.dev](https://www.11ty.dev)，MIT 许可。零配置的静态站生成器，不强制使用框架，输出纯 HTML/CSS。Astro 是另一个选择，但 11ty 更简单，更适合这种单一数据源的列表站点。
* **Lunr.js**：[lunrjs.com](https://lunrjs.com)，MIT 许可。在客户端构建全文搜索索引，无需后端搜索服务。`.brew` 文件的元数据量级（数千到数万个文件）完全在 Lunr 的舒适区。
* **ISR（增量静态再生成）**：Vercel 和 Netlify 都支持定时重新构建索引页，无需手动触发。

### 四、AI 服务层技术选型

这是 BrewCode OS 唯一需要后端计算的部分。

* **AI API：Anthropic Claude API 或 OpenAI GPT-4 API**
  * 两个都支持自然语言交互。选择取决于当时的定价、性能和上下文窗口。代码中抽象出一个 `ai_provider` 接口，随时切换。**不绑定任何单一模型供应商。**
  * 用 RAG（检索增强生成）增强：用户提问时，先从 BrewRepo 的知识库中检索最相关的 `.brew` 文件和 Issue 讨论，作为 prompt 上下文。这大幅提高诊断和生成的准确性。
* **后端运行环境：Vercel Functions（Node.js runtime）**
  * 无服务器函数。用量小时免费额度足够。每个 API 调用触发一次函数执行，冷启动延迟在 AI API 调用的耗时面前可忽略不计（AI API 本身需要 1-3 秒，冷启动 200ms 不是瓶颈）。
* **向量数据库（RAG 知识库）：LanceDB**
  * 项目地址：[lancedb.github.io/lancedb](https://lancedb.github.io/lancedb)
  * 许可证：Apache 2.0
  * 选型理由：它是目前唯一的“无服务器向量数据库”——不需要单独部署服务器，直接嵌入 Node.js 进程运行，数据存本地文件或对象存储。零运维成本，零基础设施。Pinecone/Weaviate 等托管服务要么要钱要么要服务器，不适合一个人维护的 AI 服务层。
* **隐私保护的本地执行方案：WebLLM**
  * 项目地址：[webllm.mlc.ai](https://webllm.mlc.ai)
  * 许可证：Apache 2.0
  * 这是将大模型直接跑在浏览器里的项目。当前受限于模型大小和浏览器 GPU 访问，但在诊断任务中——如果用户的设备支持 WebGPU——可以尝试在本地运行小模型完成简单诊断，数据完全不出浏览器。这是 AI 层的隐私终极方案，作为可选路径列入技术栈而非当前 MVP 依赖。

### 五、全局基础设施选型

* **域名：** `brewcode.os` 或类似。年费约 30-50 美元，唯一固定成本。这很重要——`brewcode.os/r/{user}/{repo}` 的短链跳转依赖这个域名。
* **DNS：** Cloudflare DNS。免费，全球最快之一。
* **CDN 与托管：** Vercel 或 Cloudflare Pages。两者对静态站免费，均支持无服务器函数。Vercel 的开发者体验略好，Cloudflare Pages 的全球节点更多。先用 Vercel，代码目录结构和部署命令与 Cloudflare Pages 完全兼容，迁移成本为零。
* **代码托管：** GitHub。MIT 许可证。
* **社区交流：** GitHub Issues + Discussions。不做 Slack/Discord——分散注意力的实时聊天对个人维护者是负债。

### 七、一个架构师的坦白

这个技术选型文档没有什么“前沿技术”。Alpine.js 是 jQuery 的精神继承者，CodeMirror 从 2007 年就开始开发，PouchDB 是 CouchDB 协议的浏览器实现——这些都是被时间验证过的东西。

这是刻意为之。

BrewCode OS 的长期资产不是代码，是 `.brew` 标准。代码应该尽可能薄、尽可能无聊、尽可能用别人已经踩过坑的东西。这个世界上已经有太多用最新框架重写同一个表单的“Todo App”。BrewCode OS 不做那种项目。

**选型总结为一句话：每一个依赖都必须能回答“如果它明天停止维护，我一个人能不能在两周内替换它？”——能，则入选；不能，则砍掉。**
