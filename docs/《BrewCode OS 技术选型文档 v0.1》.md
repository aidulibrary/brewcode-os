# 《BrewCode OS 技术选型文档 v0.1》

## BrewCode OS 技术选型文档 v0.1

**原则：不重新发明轮子，除非现有的轮子都是方的。**

### 零、选型铁律

在开始之前，先确立筛选标准。一个技术是否被采纳，必须同时满足以下六条：

1. **开源且许可证宽松。** MIT、Apache 2.0、BSD、CC0 优先。GPL 需评估传染性。专有软件零容忍。
2. **成熟稳定，有持续维护。** 至少有2年以上生产环境使用历史，GitHub star 数不作为主要参考，最近6个月有实质性 commit。
3. **可独立部署，不绑定特定云厂商。** 今天跑在 Vercel，明天可以迁移到 Cloudflare 或自建服务器。
4. **对个人开发者友好。** 一个人能看懂全部核心代码，文档完善，社区响应及时。
5. **极低的运行时成本。** 在日均1000次调用的规模下，成本应趋近于零。
6. **与 `.brew` JSON 标准的耦合度。** 原生支持 JSON 解析、验证、存储的技术优先。

### 一、核心复用清单：直接拿来用的

以下项目经过评估，可以直接作为 BrewCode OS 基础设施的组成部分。它们各自在某一领域做到了世界级，不需要我们再写一遍。

#### 1. GitHub —— 整个 BrewRepo 的底层存储与协作引擎

* **评级：** 开源生态的事实标准。成熟度无可匹敌。
* **许可证：** 不适用（作为平台使用，不是代码依赖）。
* **复用方式：** BrewRepo 不搭建自己的 Git 服务器，不存储文件版本历史，不实现 diff 对比。这一切由 GitHub 原生提供。
* **具体映射：**
  * 一个 `.brew` 方案 = 一个 GitHub 仓库（或一个 monorepo 中的独立目录）。
  * Fork = GitHub Fork。
  * PR = GitHub Pull Request。
  * Issue = GitHub Issues。
  * 版本历史 = Git commit log。
* **为什么不用 GitLab 或 Gitee：** GitHub 的社区网络效应无法替代。咖啡极客和开发者高度重合，他们已经有 GitHub 账号。零注册摩擦。且 GitHub 被微软收购后的十年，其开放性和稳定性已得到充分验证。国内访问偶有波动，但作为开发者工具而非面向C端消费者的产品，可接受。未来如需国内镜像，GitHub 的 API 生态允许我们随时搭建只读镜像节点。
* **我们只做一层薄薄的展示适配：** 从 GitHub API 读取 `.brew` 文件的元数据，渲染成咖啡爱好者能理解的页面，而不是代码仓库页面。这个适配层的代码量预计不超过500行。

#### 2. JSON Schema —— `.brew` 文件的验证层

* **评级：** IETF 标准，JSON 生态的基础设施。
* **许可证：** 规范本身开放。实现库各语言均有 MIT/Apache 选项。
* **复用方式：** 编写一份 `brew.schema.json`，用它来：
  * 校验用户提交的 `.brew` 文件格式是否正确。
  * 给 BrewForge 编辑器提供自动补全和字段提示。
  * 生成 API 文档（JSON Schema 可以直接渲染为人类可读的文档）。
* **为什么不用自己写验证逻辑：** 自己写意味着要处理所有边界情况、维护向后兼容性、编写错误提示。JSON Schema 的标准已经解决了这些。全球有几十个语言的成熟实现，每一个都比自己写的强。
* **具体实现：** 使用 `ajv`（JavaScript，MIT 许可证）作为浏览器端校验引擎，压缩后体积不到 50KB。在无服务器函数中用同一份 Schema 做服务端二次校验。

#### 3. Monaco Editor —— BrewForge 的代码编辑器

* **评级：** VS Code 的核心编辑器组件，微软开源，业界最成熟的浏览器内代码编辑器。
* **许可证：** MIT。
* **复用方式：** 嵌入 BrewForge，作为 `.brew` 文件的直接编辑视图。
* **为什么是它：** `.brew` 文件是 JSON。咖啡进阶用户愿意直接看和改 JSON。Monaco 提供语法高亮、括号匹配、错误波浪线、多光标编辑。这些不需要我们写一行代码。它自带 JSON 模式支持，可以直接加载我们的 `brew.schema.json` 实现智能提示——用户输入 `"ratio"` 时，编辑器自动提示字段含义和取值范围。
* **为什么不做一个纯表单编辑器而保留代码视图：** 表单是给新手用的，代码编辑器是给进阶用户用的。两者都提供，不矛盾。Monaco 的体积较大（约 5MB gzip 后约 1.5MB），但对一个专业创作工具而言可接受，且可以懒加载——用户第一次切换到“代码视图”时才下载。
* **替代方案：** CodeMirror 6（MIT）更轻量（约 200KB），如果 Monaco 的体积在实际使用中被证明是障碍，可切换到 CodeMirror。两者 API 理念相近，迁移成本可控。

#### 4. Vercel / Cloudflare Pages —— 静态托管 + 无服务器函数

* **评级：** 全球最成熟的 Jamstack 部署平台。
* **许可证：** 平台使用，非代码依赖。
* **复用方式：** BrewPlayer、BrewForge、BrewRepo 前端全部是静态页面，推送到 GitHub 后自动部署。AI 服务层用无服务器函数实现，避免自己维护服务器。
* **为什么同时保留两个选项：** Vercel 的开发体验更好（零配置部署、Serverless 函数自动扩容）。Cloudflare Pages 的全球节点更多（中国大陆访问更快），免费额度更慷慨（无带宽限制）。v0.1 从 Vercel 启动（开发速度优先），但代码本身不依赖任何 Vercel 专属 API。迁移到 Cloudflare Pages 只需改部署配置，不动业务代码。
* **为什么不买 VPS 自己部署：** 管理服务器是时间黑洞。安全更新、证书续期、进程监控、日志轮转——这些对一个人来说是不可承受的维护负担。无服务器架构把运维降到接近零。

#### 5. SQLite —— 唯一需要的数据库

* **评级：** 世界上部署量最大的数据库引擎，公共领域软件。
* **许可证：** Public Domain（无任何限制）。
* **复用方式：** BrewRepo 的元数据索引（方案列表、搜索、作者信息）存储在 SQLite 中。`.brew` 文件本体存在 GitHub。SQLite 只存索引，不存内容。
* **为什么不是 PostgreSQL 或 MySQL：** 需要一个独立的数据库服务器，意味着多一个需要监控、备份、扩展的服务。SQLite 是一个文件，零配置，零维护。对于 BrewCode OS 的数据规模（预计第一年不超过10万条方案），SQLite 的性能绰绰有余——单文件可承载 TB 级数据，每秒十万次查询。
* **具体部署：** SQLite 文件存在 Vercel 的边缘存储或 Cloudflare D1 中，挂载为只读副本供查询。写入操作通过 GitHub Action 触发重建索引——方案文件更新时，GitHub 推送事件触发 Action，运行索引脚本更新 SQLite 数据库，整体延迟不超过2分钟。对咖啡冲煮方案这种非实时场景，完全可接受。

### 二、自研决策：为什么这些必须自己写

以下组件没有现成的成熟方案，或者现有方案与我们的原则冲突，需要自己开发。

| **组件**             | **为什么不复用**                                        | **自研方案**                                              | **预估代码量** |
| ------------------ | ------------------------------------------------- | ----------------------------------------------------- | --------- |
| BrewPlayer 渲染引擎    | 现有“分步引导”类组件都是为烹饪或健身设计的，不支持 `.brew` 的时间轴+参数叠加模型    | 纯 JS，解析 JSON，渲染时序引导界面。核心是一个约200行的状态机                  | \~500行    |
| BrewForge 步骤可视化编辑器 | 需要同时输出 JSON 和实时预览 BrewPlayer 效果，市面无此组合            | 基于原生 Drag & Drop API 的轻量编辑器                           | \~800行    |
| `.brew` 到自然语言的编译器  | 这是 BrewCode OS 独有的需求：把参数文件翻译成“新手也能看懂”的冲煮说明        | 模板引擎 + 多语言映射表                                         | \~300行    |
| AI 诊断知识库的 RAG 层    | 通用 RAG 框架（如 LangChain、LlamaIndex）过重，引入了不必要的依赖和复杂度 | 简单的向量检索 + 提示词拼接，基于 Cloudflare Vectorize 或手动实现的余弦相似度匹配 | \~200行    |

**为什么要自研而不是改造现有方案：** 这四个组件是 BrewCode OS 的核心交互差异化所在。用通用框架拼凑出来的体验是“又一个普通工具”。自己写的代码量加起来不超过2000行，但能保证每一行都为咖啡冲煮场景设计。这个投入值得。

**自研部分的技术约束：** 全部用原生 JavaScript（ES2020+），不引入任何前端框架。原因：框架的折旧速度太快。今年流行的明年可能就没人维护。原生 JS 的折旧速度是零。十年后这些代码还能原样运行。

### 三、耦合设计：各组件如何连接

**1. `.brew` 文件作为唯一的数据总线。** 所有组件之间不直接通信，它们只读写 `.brew` 文件。BrewPlayer 读取它，BrewForge 编辑它，BrewRepo 存储它，AI 诊断分析它。这个设计意味着每个工具可以独立开发和部署，只要它们都遵循同一个 JSON Schema。

**2. 组件间通信使用 Web 标准。**

* BrewForge 生成 `.brew` 文件后，通过浏览器的 `postMessage` API 或简单的 URL hash 传递给 BrewPlayer 预览。
* BrewRepo 通过 GitHub REST API（官方，稳定超过15年）读写文件。
* AI 服务通过 HTTP POST 接收 `.brew` 文件内容，返回诊断结果。认证用简单的 API Key，不实现 OAuth 流程（v0.1 的用户是自己或少量信任的测试者）。

**3. 不引入消息队列、事件总线、GraphQL。** v0.1 的架构是“一个静态网站 + 几个无服务器函数 + GitHub API”。这个简单程度是刻意为之的，不是临时的妥协。BrewCode OS 的架构目标是：任何一个有基本编程能力的人，都能在一天内理解整个系统的运作方式。

### 四、语言与国际化技术选型

**问题：** BrewCode OS 的宣言和核心文档是英文的，但创始人是中文母语者，社区也必然是多语言的。

**方案：**

* **`.brew` 文件本身的字段名保持英文。** 这是技术标准，英文是编程的通用语。
* **用户界面支持多语言。** 所有面向用户的文本（按钮、提示、错误信息）用 i18n 机制管理。
* **技术选型：** 使用 `i18next`（MIT 许可证，社区最成熟的 JS 国际化框架）或其更轻量的替代 `preact-i18n`。翻译文件是简单的 JSON，社区成员可以直接提交 PR 贡献翻译。不依赖任何第三方翻译 API 做运行时翻译——成本不可控且质量不稳定。
* **AI 诊断的输出语言跟随用户输入语言。** 用户用中文描述问题，诊断结果输出中文。这个能力是 AI 模型自带的，不需要额外开发。

### 五、不选的技术及其理由

以下技术在咖啡或内容创作领域常见，但明确不采用：

* **WordPress：** 太重。为博客设计，不适合作为工具平台。插件生态复杂且安全风险高。`.brew` 文件版本管理需要 Git，不是数据库。
* **Strapi / Directus（无头 CMS）：** 过度抽象。BrewCode OS 的数据模型是单一的 `.brew` 文件，不需要关系型内容管理。引入 CMS 等于多了一个需要维护的庞大依赖。
* **Supabase / Firebase：** 优秀的后端即服务平台，但绑定特定厂商太深。如果 Supabase 修改定价或停止某个功能，迁移成本不低。我们选择 GitHub API + SQLite 的组合，这两者的替代品遍布全球。
* **React / Vue / Svelte：** 对当前项目目标而言是过度工程化。三个单页工具不需要虚拟 DOM 和状态管理库。原生 JS + Web Components 足够。如果将来有复杂交互需求（如协作冲煮实时同步），再评估框架。
* **Docker / Kubernetes：** 项目不运行任何需要容器化的服务。静态页面 + 无服务器函数不需要编排。

### 六、成本预估

**月度运营成本（第一阶段）：**

| **项目**               | **预估月费**    | **备注**           |
| -------------------- | ----------- | ---------------- |
| 域名                   | \$1.5/月     | 按年付约\$15/年，分摊到月  |
| Vercel 静态托管          | \$0         | 免费额度内（100GB带宽/月） |
| Vercel Serverless 函数 | \$0         | 免费额度内（100万次调用/月） |
| GitHub               | \$0         | 公共仓库免费           |
| AI API（诊断服务开发阶段）     | \$0         | 用个人开发者免费额度测试     |
| **合计**               | **\$1.5/月** | ​                |

**扩展到1000日活用户时：**

* 无服务器函数可能突破免费额度，按实际使用量付费。
* AI API 成本取决于诊断请求的频率。如果每次诊断成本控制在 $0.01 以内，用户付$0.50/次的情况下，毛利率约 98%。
* 即使所有免费额度用完，一个日均1000次调用的系统的月度总成本控制在 \$100 以内。这是可以被订阅收入轻松覆盖的数字。

### 七、总结

BrewCode OS 的技术选型遵循一个原则：**让系统尽可能“薄”，薄到可以被一个人完全理解和掌控。** 我们复用世界上最成熟的开源项目来完成通用任务，只在自己独一无二的需求上写代码。这个选型保证了：

* 开发速度：大部分基础设施不需要从头构建。
* 长期可维护性：没有绑定特定厂商，核心依赖都是存在十年以上且不会消失的技术。
* 极低成本：运营开支接近零，为被动收入创造了完美的成本结构。
