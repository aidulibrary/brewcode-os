**创始人角色**：标准守护者与生态架构师

### 一、创世故事

2026年，一个身在中国农村的自由人，面对三台电脑和正在学习的AI建站能力，问出一个问题：“我能否不用卖时间，而是用数字资产获得被动收入？”

在调研了图书市集、中古二手、手办、非遗、咖啡等十个领域后，一个结论浮现：咖啡是唯一将主观风味完全建立在客观物理参数上的领域，且拥有全球海量结构化公开数据。咖啡被选中。

真正改变一切的，是对问题本身的颠覆性重构。传统思维问“如何做咖啡内容网站赚流量费”，重构后的问题变成：如果风味可以被参数化，参数就可以被编程。那么咖啡的终极数字形态是什么？答案是**风味参数交易与复现系统**。它叫 BrewCode OS。

历史上有三个伟大的对应者：GitHub（代码协作与版本控制）、苹果（硬核技术的艺术封装）、IEEE（不生产商品，只定义标准）。最深刻的映射来自 Tim Berners-Lee——1989年他发明了HTTP、HTML和URL，免费开放给全世界。BrewCode OS 的使命是成为咖啡领域的万维网。

2026年某一天，BrewCode OS 官网只有三个页面：标准白皮书、BrewPlayer Lite、50个手工创建的可Fork种子文件。首页宣言只有一段话：

> “Welcome to BrewCode OS. We are building the universal language for coffee brewing. This is the first `.brew` file. Fork it.”

### 二、开发宣言

#### 第一条：基本原理

1. **咖啡是物理，不是玄学。** 任何可描述的风味差异，必然对应一组可测量的参数差异——“风味-参数同构原理”。
2. **冲煮知识应该是公共财产。** 世代传承的冲煮智慧不应被封装在专利里。
3. **工具的存在是为了被遗忘。** 最好的冲煮工具在使用时“隐退”，让人直接与咖啡对话。
4. **个体拥有数据的绝对主权。** 你的冲煮记录、味觉偏好、设备参数都属于你。

#### 第二条：核心定义

* **BrewCode OS**：为全球咖啡冲煮建立通用数字语言的开放项目。
* **`.brew`文件**：通用语言的载体，开放、JSON结构、可被任何程序读写的冲煮方案标准格式。
* **BrewPlayer**：官方运行时环境，将静态参数转化为逐步冲煮引导。
* **BrewRepo**：`.brew`文件的公共版本库，可发布、浏览、Fork、改进。

#### 第三条：社区契约

开放优先：核心标准以公共领域或MIT许可发布。协作优先：改进从Fork开始，共识通过PR达成。尊重优先：每个人的舌头都是有效传感器。隐私优先：数据采用最高标准匿名化与加密，永不售卖。

#### 第四条：商业边界

BrewCode OS 本身永远免费开源。收入仅来自可选增值服务（AI诊断、高级分析）、硬件厂商兼容认证、匿名聚合数据年度报告、社区自愿赞助。任何商业行为不得破坏标准的开放性和数据的中立性。

#### 第五条：对未来的承诺

当最后一台咖啡机停止运转，BrewCode OS 所记录的每一杯咖啡的参数和故事，将成为人类感官文化史的永久见证。我们在为人类数千年的咖啡文明建立数字图书馆。

### 三、产品架构

#### 3.1 `.brew` 标准文件结构

```json
{
  "brew": "0.1.0",
  "meta": {
    "title": "April Brewer 浅烘埃塞 经典方案",
    "author": "brewcode-os/genesis",
    "license": "CC0",
    "tags": ["light-roast", "ethiopian", "floral"]
  },
  "coffee": {
    "name": "Ethiopia Yirgacheffe Washed",
    "roaster": "Apollon's Gold",
    "roast_level": "light"
  },
  "equipment": {
    "brewer": "April Brewer",
    "grinder": "Comandante C40"
  },
  "recipe": {
    "coffee_weight_g": 12,
    "water_weight_g": 200,
    "water_temperature_c": 93
  },
  "steps": [
    {"action": "bloom", "water_g": 30, "duration_s": 30},
    {"action": "pour", "water_g": 70, "duration_s": 15}
  ],
  "result": {
    "flavor_notes": ["佛手柑", "茉莉", "柠檬酸质"],
    "body": "轻盈",
    "acidity": "明亮"
  }
}
```

**关键设计决策**：equipment用自由文本而非枚举ID——先积累数据再结构化；steps是顺序数组——冲煮是时序过程，执行顺序不可变；result可选——方案发布时可能没有测量数据；选JSON而非YAML——全球工具链最完整，零依赖解析。

#### 3.2 工具矩阵

**BrewPlayer**：给冲煮者用。载入`.brew`文件，渲染分步倒计时向导。纯前端静态页面，一个HTML文件+JS解析器，部署在Vercel/Netlify。PWA可加主屏幕，离线可用。UI极简到“扫一眼就够”。

**BrewForge**：给创作者用。可视化`.brew`创建编辑工具。包含拖拽步骤编辑器、参数校验、版本管理、一键导出。叫“编译器”因为用户输入想法和参数，输出可执行方案——这是为咖啡编程。

**BrewRepo**：给探索者用。`.brew`文件公共托管协作平台。直接映射GitHub——Repo=方案库，Fork=修改，PR=贡献，Issues=讨论。不做点赞评论关注流，协作发生在Issue里就足够。

三工具各自独立，通过`.brew`格式互通。生态入口低到只需BrewPlayer跟着方案冲煮，无需任何认知负担。

#### 3.3 AI服务层

**AI风味诊断**：用户描述问题（“这杯偏涩”），系统结合`.brew`文件输出具体调整建议。按次付费或月度订阅。核心价值是把“哪里出问题”的不确定性从几小时搜索缩减到几秒。

**AI方案生成**：输入豆子信息和设备列表，生成起点方案，精确到“用你的C40和April Brewer，针对这支浅烘埃塞，推荐研磨22格”。

**隐私设计**：能本地运行的模型优先浏览器端执行；云端需用户明确授权；匿名贡献走显式选择加入。

#### 3.4 产品边界

不做用户系统（用GitHub账号），不做社区动态推荐流，不做电商，不做移动端App（PWA足够），不做付费内容DRM，不融资。

### 四、技术选型

**选型铁律**：开源宽松许可证、成熟稳定持续维护、可独立部署不绑定云厂商、对个人开发者友好、极低运行时成本、与JSON标准耦合度高。

**核心复用**：GitHub（BrewRepo的存储与协作引擎，不自己搭Git服务器）、JSON Schema（`.brew`验证层，用ajv库）、Monaco Editor（BrewForge代码编辑器，VS Code核心组件）、Vercel/Cloudflare Pages（静态托管+无服务器函数，同时保留两个选项防锁定）、SQLite（元数据索引，Public Domain，零维护单文件数据库）。

**自研决策**：BrewPlayer渲染引擎（约500行）、BrewForge步骤编辑器（约800行）、`.brew`到自然语言编译器（约300行）、AI诊断RAG层（约200行）。全部用原生JavaScript，不引入前端框架——框架折旧速度太快，原生JS十年后还能运行。

**不选的技术**：WordPress（太重）、Strapi/Directus（过度抽象）、Supabase/Firebase（绑定厂商）、React/Vue/Svelte（过度工程化）、Docker/Kubernetes（不需要容器化）。

**月度成本**：域名$1.5/月，其余全部在免费额度内。扩展到1000日活用户时总成本控制在$100/月以内，毛利率98%。

### 五、服务分层架构

#### 5.1 理论基础

**Solid数据主权架构**（Tim Berners-Lee）：数据与应用分离，用户数据存自己控制的Pod，应用只是临时授权访问。BrewCode OS不拥有任何用户数据——这一原则来自Solid。

**无服务器边缘计算服务网格**：服务治理逻辑从应用代码剥离到基础设施层。BrewCode OS用一个约100行的网关函数实现路由、鉴权、限流，复用Cloudflare全球边缘网络。

**关联数据与语义网**：数据携带语义标注后，机器可理解其含义。`.brew`文件中“Comandante 22格”将被映射为“约800μm粒径”，实现跨设备参数翻译。

#### 5.2 五层架构

**第一层 基础设施**：Cloudflare Pages/Vercel + GitHub + 互联网根基础设施。总依赖项不到10个，一个下午即可完成平台迁移。

**第二层 数据与标准**：`.brew` JSON Schema是数据格式宪法。文件本体存GitHub，元数据索引在SQLite用FTS5全文搜索。语义映射表由社区维护的YAML文件定义。

**第三层 业务服务**：五个无服务器函数——AI诊断、方案生成、语义翻译、索引搜索、Webhook处理。每个独立部署独立扩缩，无状态，30秒超时。

**第四层 API网关**：单一Cloudflare Worker，统一处理鉴权（API Key）、限流（每分钟60次）、路由、版本管理。API路径含版本号，旧版本保持12个月后退役。

**第五层 交互与分发**：纯静态资源通过全球CDN分发，PWA离线可用。版本化URL保证旧链接不会因升级而失效。

**不引入**：容器编排、独立认证服务（GitHub OAuth+30行KV查询足够）、微服务编排引擎、专属数据分析管道。

### 八、风险清单

**风险1：标准推广失败。** 最致命的风险。`.brew`文件未被足够多人使用，生态无法启动。缓解：前50个种子方案覆盖主流器具和豆子，确保早期访问者能找到与自己相关的方案；AI方案生成降低创作门槛；创始人亲自在咖啡社区逐个邀请早期用户。

**风险2：AI服务体验不佳。** 诊断不准，推荐泛泛，用户不愿付费。缓解：严格控制初期种子方案质量；诊断结果必须落到具体操作层面而非泛泛建议；早期免费收集反馈快速迭代；若通用模型不够好，用高质量方案日志做微调。

**风险3：被大公司复制。** 巨头看到验证的模式后推出类似标准。缓解：速度是最强护城河——大公司从决策到执行需6-12个月，这是BrewCode OS建立网络效应的窗口期；中立性是结构优势，巨头有利益冲突；标准先发优势一旦形成，后来者用标准锁定效应对抗。

**风险4：创始人精力耗尽。** 一个人维护所有代码、内容、社区、商业。缓解：极致自动化（索引重建、方案校验、CI/CD全自动）；GitHub协作天然分布式；收入足够后优先委托非核心任务（文档翻译、社区管理）。

**风险5：法律与合规。** `.brew`文件可能包含他人版权内容；AI诊断建议若导致用户使用不当（如烫伤）可能引发责任争议。缓解：标准文档声明CC0许可，Repo默认MIT；AI服务添加免责声明（“此建议为辅助参考，冲煮请遵循安全操作规范”）。
