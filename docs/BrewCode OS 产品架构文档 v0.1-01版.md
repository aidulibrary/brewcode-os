### BrewCode OS 产品架构文档 v0.1

写完宣言之后，接下来要回答一个问题：这个东西到底包含哪些产品，它们怎么组合在一起。

先说一个核心原则：**BrewCode OS 不做“大而全”的平台，不做咖啡版的小红书，不做豆子电商。** 我们做的是三个东西的精确组合：一个开源标准、一套围绕标准的工具、一层跑在工具之上的AI服务。这三层互不绑定，但合在一起构成完整闭环。

***

#### 第一层：`.brew` 标准文件

这是整个生态的原子单位。没有这个，后面的一切都不成立。

**它是什么：** 一个 JSON 文件，描述“如何冲一杯咖啡”所需的全部参数。

**它的结构（v0.1）：**

```json
{
  "brew": "0.1.0",
  "meta": {
    "title": "April Brewer 浅烘埃塞 经典方案",
    "author": "brewcode-os/genesis",
    "source": "https://brewcode.os/r/genesis/001",
    "license": "CC0",
    "created": "2026-06-12",
    "tags": ["light-roast", "april-brewer", "ethiopian", "floral"]
  },
  "coffee": {
    "name": "Ethiopia Yirgacheffe Washed",
    "roaster": "Apollon's Gold",
    "roast_date": "2026-05-20",
    "roast_level": "light",
    "varietal": "Heirloom",
    "process": "washed"
  },
  "equipment": {
    "brewer": "April Brewer",
    "filter": "April paper filter",
    "grinder": "Comandante C40",
    "kettle": "Fellow Stagg EKG"
  },
  "recipe": {
    "coffee_weight_g": 12,
    "water_weight_g": 200,
    "ratio": "1:16.67",
    "water_temperature_c": 93,
    "grind_setting": "Comandante 22 clicks"
  },
  "steps": [
    {"action": "rinse_filter", "note": "用热水冲洗滤纸并预热分享壶"},
    {"action": "bloom", "water_g": 30, "duration_s": 30, "note": "中心注水，确保所有粉湿润"},
    {"action": "pour", "water_g": 70, "duration_s": 15, "note": "绕圈注水至100g"},
    {"action": "pour", "water_g": 100, "duration_s": 15, "note": "中心注水至200g"},
    {"action": "drawdown", "target_s": 150, "note": "总萃取时间约2分30秒"}
  ],
  "result": {
    "tds": 1.32,
    "extraction_yield": 20.4,
    "flavor_notes": ["佛手柑", "茉莉", "柠檬酸质", "绿茶余韵"],
    "body": "轻盈",
    "acidity": "明亮",
    "sweetness": "中等"
  }
}
```

**设计决策记录：**

* **`equipment` 用字符串不用枚举ID？** 对，v0.1先这样。枚举ID要求有一个被全球认可的“设备注册表”，这需要时间建立。先用自由文本，等数据量足够大时再迁移到结构化ID。
* **`steps` 是顺序数组而非对象？** 对，因为冲煮是一个时序过程。数组保证了执行顺序，这是关键。brew不关心你在第几步做了什么花样，但必须知道先做什么后做什么。
* **`result` 是可选的？** 对。方案发布时可能没有测量数据，但冲完之后可以补充。这是数据的“方案层”和“记录层”分离。
* **为什么不是 YAML？** YAML 对非程序员更友好，但 JSON 的生态工具链是全球最完整的。任何语言、任何平台都能零依赖解析 JSON。我们选择JSON是出于兼容性，不是审美偏好。

**这个文件的能力边界：** 它能描述一杯手冲咖啡的完整冲煮方案。它不能描述意式浓缩（需要压力曲线，那是另一个数据结构），不能描述冷萃（时间维度完全不同）。这些是未来扩展的方向，不是v0.1要解决的问题。

#### 第二层：工具矩阵

工具是让 `.brew` 文件从“一段文本”变成“一杯咖啡”的东西。三类用户需要三种工具。

**工具一：BrewPlayer（播放器）**

* **给谁用：** 只想跟着方案冲出一杯好咖啡的人。
* **做什么：** 载入 `.brew` 文件，把它渲染成一个分步倒计时的冲煮向导。
* **界面：** 极简。顶部是方案名和作者，中间是当前步骤的大字提示（“绕圈注水至100g”），一个倒计时条，一个下一步按钮。没了。
* **设计原则：** 冲煮时人的注意力在咖啡上，不在手机上。UI 必须做到“扫一眼就够”。颜色、动效、品牌标识全部克制。
* **技术选型：** 纯前端静态页面，一个 HTML 文件加一个 JS 解析器。部署在 Vercel/Netlify。不需要后端，不需要数据库，不需要用户系统。
* **为什么不做 App：** v0.1不需要。PWA（渐进式网页应用）可以加到主屏幕，体验接近原生。App的审核、更新、双平台开发成本完全不值得。

**工具二：BrewForge（编译器）**

* **给谁用：** 想自己设计、修改、测试冲煮方案的人。
* **做什么：** 一个可视化的 `.brew` 文件创建和编辑工具。左侧是表单+文本编辑器，右侧实时预览成 BrewPlayer 的样子。
* **关键功能：**
  * 可视化步骤编辑器（拖拽排序、增删步骤）
  * 参数校验（“粉水比低于1:15，这杯可能偏浓”）
  * 版本管理（每次保存生成新版本，可以回退）
  * 一键导出 `.brew` 文件或分享链接
* **为什么叫编译器：** 因为用户的输入是“想法和参数”，输出是“可执行的冲煮方案”。这个过程本质上就是编译。名字暗示了它的严肃性——这不止是填表，这是为咖啡“编程”。

**工具三：BrewRepo（仓库）**

* **给谁用：** 想发现新方案、分享自己方案、或参与协作的人。
* **做什么：** `.brew` 文件的公共托管和协作平台。
* **核心机制直接映射 GitHub：**
  * **Repo = 一个咖啡方案库**（可以是个人合集、某个豆子的多方案、某个设备的所有适配方案）
  * **Fork = 拿走别人的方案，按自己的设备和口味修改**
  * **Pull Request = 你的修改经原作者审核后合入主线**
  * **Issues = 讨论这个方案哪里好、哪里可以改进**
* **为什么不做社交功能：** 点赞、评论、关注、推荐流——这些都是噪音。BrewRepo 只做一件事：让参数文件的协作尽可能简单。讨论发生在 Issue 里，足够。
* **搜索维度：** 按豆种、烘焙度、器具、研磨机、风味标签。这是五个最关键的检索入口。

**工具之间的关系：**&#x42;rewForge 创建的文件推送到 BrewRepo。BrewRepo 上的文件可以在 BrewPlayer 里一键打开。三个工具各自独立，但通过 `.brew` 文件格式互通。没有人必须三个都用。你可以只用 BrewPlayer 跟着别人的方案冲煮，永远不碰另外两个。这很重要——**生态的入口必须足够低，低到没有任何认知负担。**

#### 第三层：AI服务

工具层解决“如何做”，AI层解决“为什么”和“怎么更好”。

**服务一：AI风味诊断（面向消费者的核心付费产品）**

* **用户输入：** 自然语言描述一个问题或一杯咖啡的感受。
* **系统输入（如果用户授权）：** 用户的历史冲煮数据、设备信息、最近使用的方案。
* **输出：** 具体到操作层面的调整建议。“把研磨调粗半格（Comandante从22调到23），水温降低2°C到91°C，闷蒸时间缩短5秒。”
* **训练数据来源：** BrewRepo上所有CC0和MIT许可的方案及其关联的冲煮记录；公开的咖啡萃取研究论文数据；社区中标记为“已解决”的Issue讨论。
* **为什么用户愿意付费：** 因为一个咖啡爱好者在冲出一杯“不对”的咖啡时，最大的痛苦不是难喝，是不知道哪里出问题了。他搜小红书，答案互相矛盾。他问朋友，朋友没喝过这支豆子。AI诊断把这个不确定性从“几小时搜索”缩减到“几秒钟”。
* **价格：** v0.1按次付费，一次诊断等于一杯好咖啡的钱。这是用户心理账户中最容易接受的等价物。

**服务二：AI方案生成（面向进阶用户的增值服务）**

* **输入：** 一支豆子的信息（产地、烘焙度、处理法）+ 用户的设备列表 + 口味偏好。
* **输出：** 一个可直接载入 BrewPlayer 的 `.brew` 文件初稿。
* **核心能力：** 不是泛泛的“推荐手冲参数”，而是精确到“用你的C40和April Brewer，针对这支浅烘水洗埃塞，推荐一个起点方案，并预测可能出现的问题”。
* **与社区方案的区别：** 社区方案是“这是别人觉得好喝的”。AI方案是“基于你的设备和这支豆子的特性，理论上最合理的起点”。两者互补。

**AI层的隐私设计（关键）：**&#x51B2;煮数据包含个人信息：你有什么设备、你喜欢什么口味、你什么时候在家。这些数据不能离开用户的控制。

* 本地诊断：能跑在浏览器里的模型优先在本地运行。
* 云端可选：复杂诊断需要更强模型时，明确告知并让用户选择。
* 匿名贡献：如果用户愿意让数据用于改进公共模型，必须走显式的“选择加入”，数据脱敏后聚合使用。
* **一个承诺：** BrewCode OS 永远不卖用户的冲煮数据。这不是因为高尚，是因为只要卖了一次，信任就消失了，而信任是这个生态唯一真正的护城河。

#### 产品架构总览

```
┌─────────────────────────────────────┐
│            AI 服务层                │
│  ┌─────────────┐ ┌───────────────┐  │
│  │ 风味诊断 API │ │ 方案生成 API  │  │
│  └─────────────┘ └───────────────┘  │
├─────────────────────────────────────┤
│            工具层                   │
│  ┌──────┐ ┌────────┐ ┌──────────┐  │
│  │Player│ │ Forge  │ │  Repo    │  │
│  │ 播放  │ │ 编译器  │ │  仓库    │  │
│  └──────┘ └────────┘ └──────────┘  │
├─────────────────────────────────────┤
│            标准层                   │
│  ┌──────────────────────────────┐  │
│  │        .brew JSON 标准       │  │
│  └──────────────────────────────┘  │
├─────────────────────────────────────┤
│            基础设施                 │
│  ┌──────────────────────────────┐  │
│  │   GitHub / Vercel / CDN      │  │
│  │   静态托管 + 无服务器函数     │  │
│  └──────────────────────────────┘  │
└─────────────────────────────────────┘
```

**技术选型记录：**

* **前端：** 纯 HTML/CSS/JS，不用框架。三个工具都是单页应用，体积控制在50KB以内。用框架的人不理解为什么要不用框架。理由：十年后这些工具还能打开。不依赖npm，不依赖构建工具链。
* **后端（仅在AI层需要）：** 无服务器函数（Vercel Functions / Cloudflare Workers）。冷启动延迟在AI调用的上下文中可接受。不需要维护服务器。
* **数据库：** BrewRepo的元数据存在SQLite里，文件本体存在GitHub Repo的`/brews`目录下。GitHub本身就是最可靠的数据库。
* **AI模型：** 诊断和生成用现有大模型API（如Claude/GPT），辅以BrewCode OS自己的知识库做RAG（检索增强生成）。未来有足够自有数据时考虑微调一个小模型。
* **一个关键选择：为什么不用Supabase/Firebase：** 它们是优秀的产品。但BrewCode OS的架构故意避免绑定任何特定云服务商。一个静态网站+GitHub+无服务器函数，可以在任何地方部署，甚至可以在一台树莓派上运行全部功能。这是为长期可维护性做出的选择——**这个项目应该能活到Cloudflare和Vercel都不存在的那一天。**

***

#### 不做什么（v0.1明确排除）

* **不做用户系统。** BrewPlayer和BrewForge不需要登录。BrewRepo用GitHub账号登录（因为它本质上是GitHub的一个镜像层）。自己的用户系统是负债，不是资产。
* **不做社区、动态、推荐流。** 所有“社区”功能通过GitHub Issues和Pull Requests承载。内容分发靠搜索引擎和`.brew`文件自身的可分享性。
* **不做电商。** 不卖豆子，不卖器材，不接入商品链接。一次也不。
* **不做移动端App。** PWA足够。App Store和Google Play的审核流程、开发成本、更新节奏，都是一个人无法承受的。
* **不做付费内容的DRM（数字版权保护）。** `.brew`文件是纯文本。如果有人把付费方案免费分享出去，那是社区治理问题，不是技术问题。加密只会伤害诚实的用户。
* **不融资。** BrewCode OS的运营成本极低（域名、无服务器函数的少量费用）。如果有收入（AI服务、认证费），它会自给自足。外部资本的需求等于零，外部资本引入的控制权则是致命的。

***

#### 一句话总结产品策略

**BrewCode OS 是一个极薄的标准层加极轻的工具层，它的价值不在于代码量，而在于它定义的协议被多少人使用。产品做减法，生态做加法。**
