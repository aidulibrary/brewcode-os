## BrewCode OS 平台服务分层设计文档 v0.1

### 零、分层总览

BrewCode OS 的服务架构分为四层。从下往上，每一层都只依赖下一层，不知道上一层的存在。

```
┌─────────────────────────────────────────────────┐
│              第四层：AI 服务层                    │
│  风味诊断引擎 · 方案生成器 · 知识库 RAG          │
├─────────────────────────────────────────────────┤
│              第三层：工具与应用层                 │
│  BrewPlayer · BrewForge · BrewRepo 展示层        │
├─────────────────────────────────────────────────┤
│              第二层：存储与分发层                 │
│  GitHub Repos · CDN · SQLite 索引                │
├─────────────────────────────────────────────────┤
│              第一层：标准与协议层                 │
│  .brew JSON Schema · 风味标签体系 · 设备标识规范  │
└─────────────────────────────────────────────────┘
```

**分层原则：**

* **每一层可以独立替换。** 如果将来不用 GitHub 了，只改第二层，第三层无感知。
* **每一层有自己的“公共接口”。** 第一层的接口是 JSON Schema。第二层的接口是文件读写 API。第三层的接口是 HTTP 页面和无服务器函数。第四层的接口是 AI API。
* **数据永远向下引用，事件向上传递。** 工具层不直接写 AI 层的数据，它只向上发送请求。AI 层不直接操作存储层，它通过工具层间接获取数据。

### 一、第一层：标准与协议层

这是 BrewCode OS 的宪法。没有这一层，上面的所有东西都是沙上的城堡。

#### 1.1 `.brew` JSON Schema

* **定位：** 咖啡冲煮方案的唯一机器可读标准格式。
* **当前版本：** v0.1.0
* **文件：** `brew.schema.json`
* **维护方式：** 存放在 `github.com/brewcode-os/spec` 仓库中。版本号遵循语义化版本。任何结构性变更必须经过公开 RFC（征求意见）流程。
* **兼容性承诺：** v0.x 期间不保证向后兼容。v1.0.0 起，主版本号不增加则旧文件永不被破坏。这是从语义化版本标准中直接继承的成熟做法，不需要我们自己发明规则。
* **Schema 的核心结构已在产品架构文档中定义。** 这里补充的是“标准层的治理”而非 Schema 本身的内容。

#### 1.2 风味标签体系

* **问题：** 咖啡风味描述极其混乱。有人写“柠檬酸”，有人写“柑橘酸质”，有人说“酸感明亮”。AI 无法理解这些是同义词。
* **方案：** 建立一个 **三层的风味标签树**。
  * **第一级：大类**（酸、甜、苦、体脂感、余韵）
  * **第二级：子类**（柑橘类酸、莓果类酸、核果类酸……）
  * **第三级：具体风味**（柠檬、佛手柑、橙子……）
* **数据来源：** 基于 SCA（精品咖啡协会）风味轮进行简化和扩展，加入亚洲咖啡特有的风味词（如“乌龙茶感”、“红枣”、“龙眼”）。SCA 风味轮是行业事实标准，直接基于它扩展可以避免重新造轮子的同时保持与行业术语的兼容性。
* **JSON 表示：** 在 `.brew` 文件的 `result.flavor_notes` 中使用标签树中的 ID，而非自由文本。例如 `"flavor_notes": ["flavor:bergamot", "flavor:jasmine", "flavor:lemon-acidity"]`。
* **兼容期：** v0.1 阶段允许同时填写自由文本和标签 ID。工具层负责将自由文本映射到标签（通过简单的同义词表），映射不了的保留原文并标记“未分类”。这个“未分类”清单本身就是标签体系迭代升级的最重要输入源。

#### 1.3 设备标识规范

* **问题：** “Comandante C40”有人写全称，有人写“C40”，有人写“司令官”。这和风味标签的问题是同源的。
* **方案：** 维护一个 `equipment.json` 的设备注册表。
  * 每种设备有唯一 ID：`grinder:comandante-c40`
  * 有品牌、型号、类型（手摇/电动）
  * 有常用的别名列表（`"aliases": ["C40", "司令官C40"]`）
* **注册方式：** 社区通过 PR 提交新设备。核心团队（最初就是你）审核合并。
* **为什么不自动抓取电商数据：** 电商数据需要清洗和维护，且涉及阿里系平台（这是你明确排除的）。社区手工提交虽然慢，但数据质量高，且让贡献者有参与感。

#### 1.4 标准层与上层的关系

* 第一层只输出一个东西：**规范文件。**
* 第二层负责存储这些规范文件和遵循这些规范的 `.brew` 实例。
* 第三层负责校验用户提交的 `.brew` 文件是否符合规范。
* 第四层负责基于这些结构化数据提供智能服务。

### 二、第二层：存储与分发层

这一层负责“数据放在哪里、怎么访问”。它不关心数据的内容，只关心数据的存储和传输。

#### 2.1 GitHub 作为主存储

**存储结构：**

```
github.com/brewcode-os/
├── spec/                    # 标准规范
│   ├── brew.schema.json
│   ├── flavor-taxonomy.json
│   └── equipment.json
├── data/                    # .brew 文件公共数据集
│   ├── recipes/
│   │   ├── v60/
│   │   │   ├── 4-6-method.brew
│   │   │   └── osmotic-flow.brew
│   │   └── april-brewer/
│   │       └── light-roast-ethiopian.brew
│   └── index.sqlite         # 自动生成的元数据索引
├── website/                 # 官网和工具源代码
│   ├── player/
│   ├── forge/
│   └── repo-explorer/
└── rfcs/                    # 标准变更提案
```

**设计理由：**

* **一切公开。** 没有私有仓库，没有内部数据。从第一天起就是完全透明的公共基础设施。
* **Git 历史就是审计日志。** 谁在什么时候改了什么，不需要额外记录。
* **Fork 就是备份。** 每个 Fork 了 `data` 仓库的人，都持有一份完整的 BrewCode OS 数据副本。即使原始仓库消失，数据不会丢失。

#### 2.2 CDN 作为分发加速层

* **为什么需要：** GitHub 的原始文件访问有速率限制，且国内访问不稳定。
* **方案：** 用 Vercel/Cloudflare Pages 部署时，静态资源（包括 `.brew` 文件）自动分发到全球 CDN。
* **访问路径：**
  * 开发者/工具通过 GitHub API 获取文件（权威数据源）。
  * 普通用户通过 `brewcode.os/r/...` 访问 CDN 缓存的副本（快速、不计 API 限额）。
* **一致性保证：** GitHub 是单一数据源。CDN 是只读缓存。不存在“CDN 上的数据和 GitHub 不一致”的可能——缓存过期或 GitHub webhook 触发重建后，CDN 总是从 GitHub 重新拉取。

#### 2.3 SQLite 索引层

* **问题：** GitHub 的文件树不支持复杂的查询。“找出所有用 V60 冲煮浅烘埃塞的方案”——这需要对所有 `.brew` 文件做全文检索，每次查询都遍历整个仓库是不现实的。
* **方案：** 一个自动生成的 SQLite 数据库，只存储 `.brew` 文件的元数据：
  * 文件名、路径、作者
  * `coffee.name`、`coffee.roast_level`
  * `equipment.brewer`、`equipment.grinder`
  * `result.flavor_notes`（标签化后的）
  * 创建时间、最后修改时间
* **更新机制：** GitHub Action 在 `data` 仓库收到 push 时触发，扫描所有 `.brew` 文件，提取元数据，重建 `index.sqlite`，提交回仓库。整个流程从 push 到索引更新完成不超过两分钟。
* **访问方式：** 工具层不直接查询 SQLite 文件。一个无服务器函数接收查询请求，在后端读取 SQLite，返回 JSON 结果。对工具层而言，这就是一个普通的搜索 API。

### 三、第三层：工具与应用层

这一层是用户直接接触的部分。三个核心工具加一个展示站。

#### 3.1 BrewPlayer（方案播放器）

* **定位：** 冲煮时的实时向导。用户从 BrewRepo 或朋友那里得到一个 `.brew` 文件，在 BrewPlayer 中打开，然后跟着做。
* **输入：** `.brew` 文件的 URL 或直接粘贴的 JSON 文本。
* **输出：** 一个分步骤、带计时器的冲煮指导界面。
* **技术实现：**
  * 纯静态页面，托管在 CDN。
  * 所有解析和渲染在浏览器中完成，不向服务器发送任何数据。
  * 支持 PWA（渐进式网页应用），可添加到手机主屏幕，离线使用。
* **关键交互细节：**
  * 当前步骤用大字体显示（“绕圈注水至 100g”），以便在冲煮时一瞥即得。
  * 计时器有视觉进度条，不需要精确看数字就能感知时间。
  * 下一步按钮做得足够大，适合湿手操作（这是咖啡冲煮中的真实场景）。
  * 完成冲煮后，弹出可选输入框：“实际风味如何？”用户填写的反馈可以保存为个人记录，也可以选择匿名提交到公共数据集。这个反馈回路是后续 AI 诊断训练数据的来源。

#### 3.2 BrewForge（方案编译器）

* **定位：** 方案创作工具。给那些不满足于使用别人方案、想要自己设计和分享方案的人。
* **输入：** 用户通过表单或代码编辑器创建/修改的冲煮方案。
* **输出：** 符合 `.brew` Schema 的 JSON 文件。
* **双视图设计：**
  * **表单视图：** 给不想看代码的用户。下拉选择设备，滑块调整参数，拖拽排布步骤。
  * **代码视图：** Monaco Editor，直接编辑 JSON。自动补全、实时校验、错误提示。
  * 两个视图实时同步。在表单中修改一个值，代码视图中对应字段立即更新，反之亦然。
* **内置校验：**
  * 保存时自动运行 JSON Schema 校验。
  * 逻辑校验：粉水比是否合理（低于 1:13 或高于 1:20 给出警告）、闷蒸水量是否约为粉重的2-3倍、总萃取时间是否在常见范围内。
  * 校验规则设为“警告”而非“错误”——超出常规范围的参数可能是刻意为之的创新，不应该被工具阻止，但需要提醒创作者注意。

#### 3.3 BrewRepo 展示层（方案发现平台）

* **定位：** 公共方案库的浏览器。不做社交，只做搜索和发现。
* **关键页面：**
  * **首页：** 最近更新的高质量方案（人工或算法筛选）。
  * **搜索页：** 按豆种、器具、研磨机、烘焙度、风味标签五个维度组合筛选。
  * **方案详情页：** 展示一个 `.brew` 文件的全部内容，用人类可读的格式，而非 JSON 树。页面底部有“在 BrewPlayer 中打开”、“在 BrewForge 中编辑”、“Fork 此方案”三个按钮。
  * **作者页：** 一个人发布的所有方案列表。
* **社交功能的最小化实现：**
  * “Star” = 收藏。
  * “Fork” = 跳转 GitHub。
  * “讨论” = 跳转 GitHub Issues。
  * 没有点赞数展示在显著位置。方案的质量不由点赞数定义，由实际冲煮体验定义。展示层只呈现信息，不呈现社交博弈。

#### 3.4 官网与文档站

* **定位：** BrewCode OS 的门面。
* **内容：**
  * 宣言
  * 标准文档
  * 工具使用指南
  * 贡献指南（如何提交设备、风味标签、方案）
  * RFC 列表和流程说明
* **实现：** 静态站点，Markdown 写作，GitHub 驱动。用 VitePress 或 Starlight 生成（两者都是 MIT 许可证，专为文档站设计，零运行时 JavaScript 成本）。

### 四、第四层：AI 服务层

这是 BrewCode OS 的商业引擎，也是唯一有运营成本的层。

#### 4.1 风味诊断引擎

* **定位：** 为咖啡爱好者提供具体、可操作的风味问题诊断。
* **输入：**
  * 用户的自然语言描述（“这杯偏苦，而且喝完后嘴里很干”）
  * 本次使用的 `.brew` 文件（如果用户提供）
  * 用户授权访问的历史冲煮记录（如果提供）
  * 用户设备信息（如果提供）
* **处理流程：**
  1. **语义解析：** AI 提取用户描述中的关键信息（“苦”+“干/涩”）。
  2. **参数比对：** 如果提供了 `.brew` 文件，AI 检查参数是否有明显问题（研磨太细？水温过高？粉水比偏低？）。
  3. **知识库检索（RAG）：** 在 BrewCode OS 的已知问题-解决方案库中搜索类似案例。
  4. **生成诊断结果：** 给出 1-3 个可能原因，每个原因对应一个具体的调整建议。
* **输出格式：** 不是一篇论文，而是三行以内的结论。“最可能的原因是研磨偏细。建议调粗1格（C40从22到23），其他参数不变。如果下次还苦，再降低2°C水温。”
* **关键设计约束：** AI 在给出建议时必须提供可验证的预测——“调粗研磨后，苦味应该减弱，甜感会更突出”。这使用户能够在下次冲煮时验证建议是否正确，错误的诊断会被用户反馈标记，进入模型改进循环。

#### 4.2 方案生成器

* **定位：** 帮助用户为特定豆子找到最佳起始参数。
* **输入：** 豆子信息 + 用户设备 + 口味偏好。
* **处理流程：**
  1. **匹配已有方案：** 在 BrewRepo 中搜索相似豆种+相同器具的方案。如果找到高评分方案，直接推荐并说明“社区中 XX 人用类似设置冲出了满意的结果”。
  2. **生成新方案：** 如果无匹配，AI 基于该豆种的特性（产地、处理法、烘焙度）和用户设备，生成一个起点方案。生成逻辑参考了公开的冲煮指南知识（如 Barista Hustle、Jonathan Gagné 的博客等已经被社区广泛验证的方法论）。
  3. **标注不确定性：** 对 AI 最不确定的变量（如“这支豆子的最佳研磨度”）主动提示：“研磨度建议从 C40 22格开始试，如果偏苦就调粗，偏酸就调细。”
* **输出：** 一个完整的 `.brew` 文件，加上一段“调整指南”。

#### 4.3 AI 层的技术实现

* **模型选择：** 使用成熟的 LLM API（Claude / GPT）。不自己训练模型。我们的价值不在模型本身，而在提示词工程和知识库的质量。
* **知识库构建：**
  * **种子数据：** 咖啡科学公开论文的摘要、Barista Hustle 等权威博客的文章（取得授权或仅使用公开知识）。
  * **社区数据：** BrewRepo 中标记为“已验证”的方案（作者自己冲过并满意）及其冲煮记录。
  * **反馈循环：** 用户对诊断结果的“有用/无用”反馈直接标记在对应的知识条目上，低评分条目定期从检索库中降权或移除。
* **成本控制：**
  * 缓存：相似问题（“我的 V60 偏苦怎么办”）直接返回缓存结果，不调用 AI。
  * 提示词优化：精简系统提示，减少不必要的 Token 消耗。
  * v0.1 阶段用免费额度覆盖测试用户，进入付费阶段后每次诊断的成本已经包含在定价中。

#### 4.4 隐私与数据策略

* **AI 诊断的隐私模型：**
  * **无账号模式：** 用户不需要注册就能使用诊断。输入问题和参数，得到建议。数据不留存。
  * **账号模式（可选）：** 用户可以创建账号，授权诊断引擎访问其历史记录以获得更精准的建议。历史数据加密存储，用户可随时导出或删除。
  * **数据用于改进模型：** 必须通过显式的“选择加入”。默认不收集。这限制了模型改进的速度，但这是信任的必要代价。

### 五、跨层服务：认证与计费

这部分不属于任何一层，但贯穿整个系统。

#### 5.1 认证

* **v0.1 策略：不要求强制认证。**
* BrewPlayer：完全匿名，不需要登录。
* BrewForge：本地工具，不需要登录。导出方案时才需要选择发布到 BrewRepo（此时需 GitHub 账号）。
* BrewRepo：浏览不需要登录。发布方案需要 GitHub 账号。Fork 需要 GitHub 账号。
* AI 诊断：免费额度不需要登录。高频使用或高级功能需要创建账号。
* **账户系统选择：** 只支持 GitHub OAuth 登录。不做自己的用户名/密码系统。理由：我们的目标用户是咖啡极客和开发者，他们已经有 GitHub 账号。自己做账户系统意味着要处理密码重置、安全漏洞、多因素认证——这是无尽的安全债务。

#### 5.2 计费

* **v0.1 阶段：免费。** AI 诊断用开发者的个人 API 额度覆盖早期测试用户。
* **v1.0 后计费模型：**
  * **AI 诊断：** 按次或按月订阅。
  * **方案生成：** 包含在订阅中。
  * **BrewCode 兼容认证（B端）：** 对硬件厂商收取认证费用。
* **支付集成：** 用 Stripe（全球）或 Lemon Squeezy（对个人开发者更友好的 MoR，处理全球税务）。不碰支付信息的存储。国内支付方案未来根据用户需求评估，但 v1.0 不引入额外复杂度。

### 六、服务等级与 SLA

**坦诚声明：** BrewCode OS 是一个个人领导的社区项目。v0.1 阶段不承诺任何 SLA（服务等级协议）。

* **工具的可用性取决于：** GitHub 是否在线，Vercel/Cloudflare 是否在线。这两个平台的历史可用性都超过 99.9%，对咖啡冲煮场景足够。
* **数据安全取决于：** GitHub 的安全和你的 Git 仓库设置。
* **AI 服务质量取决于：** 所用 AI 模型提供商的可用性。如果 Claude API 挂了，诊断服务会返回降级提示而非错误信息——“诊断服务暂时不可用，请稍后重试。”

**这不是一个需要 24/7 在线的系统。** 没有人会因为晚 30 分钟获得咖啡诊断而遭受损失。这种天然的容忍度让我们可以用极简的架构支撑一个全球服务。
