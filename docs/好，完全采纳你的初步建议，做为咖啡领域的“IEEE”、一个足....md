## 第二部分：BrewCode OS 项目开发宣言

### 2.1 总则

BrewCode OS 不是一个公司，不是一个App，不是一个电商平台。

它是一个**社会化开放协议**，一个**参数文件标准**，一个**为全球咖啡冲煮领域提供底层数字基础设施的公共物品**。

我们相信：一杯咖啡的风味，是其物理参数的必然结果，而非玄学。

我们相信：冲煮知识不应被少数专家垄断，而应成为全球爱好者共同贡献、验证、迭代的公共代码。

我们相信：定义规则的权利，属于每一个愿意贡献的个体，而非商业巨头。

### 2.2 哲学纲领

**公理一：风味-参数同构原理**任何可描述的风味差异，必然对应一组可测量的参数差异。主观的“好喝”与“难喝”，根源于客观的萃取动力学。

**公理二：工具隐去，参数涌现**最好的工具，让使用者意识不到工具的存在。BrewCode OS 的设计目标，是让参数界面成为感官的延伸，而非认知的障碍。

**公理三：冲煮自由是代码自由**每个人有权自由获取、修改、分享使其咖啡更美味的“源代码”（`.brew`文件）。我们反对任何形式的参数封闭与知识垄断。

### 2.3 生态位定义

**一句话定位：** 咖啡冲煮领域的底层数字操作系统与参数标准制定者。

**三维映射：**

* **GitHub 的协作架构**：参数仓库、Fork/Pull Request机制、版本控制。
* **IEEE 的标准权威**：`.brew`文件格式规范、术语表、互操作性认证。
* **苹果的体验封装**：极简交互界面，将复杂萃取科学封装为直觉操作。

**核心信条：** 我们不卖咖啡豆，不卖冲煮设备，不卖广告。我们只做一件事——定义和提供通用的冲煮语言。正是这种极致的“不做”，构成了我们最坚固的护城河。

### 2.4 开源许可与社区准则

**许可协议：**

* `.brew`文件格式规范：采用知识共享 CC BY-SA 4.0（署名-相同方式共享）。
* 核心代码库（BrewPlayer, BrewForge等）：采用 MIT 许可证。
* 数据（社区贡献的`.brew`文件库）：采用开放数据共享公共领域贡献与许可（ODC PDDL）。

**社区准则（初版）：**

1. **尊重原创**：Fork时保留原作者署名。
2. **可复现为先**：发布的方案必须经过实际测试，标注测试环境（设备、水质大致范围）。
3. **善意协作**：Pull Request 讨论聚焦于参数与数据，禁止人身攻击。
4. **开放优于封闭**：默认公开发布，除非涉及商业敏感信息（如未发布的烘焙配方）。

### 3.2 工具矩阵

**A. BrewPlayer（冲煮播放器）**

* **功能**：加载`.brew`文件，将其转化为分步引导的冲煮流程。界面包含计时器、注水指示、目标流速提示。
* **定位**：面向普通用户的入口产品。免费。
* **MVP形态**：一个单一网页应用，粘贴JSON即可启动。
* **技术实现**：纯前端，HTML+JS，AI可生成。
* **英文名称**：BrewPlayer
* **中文名称**：BrewPlayer Lite（保留英文，简洁）

**B. BrewForge（方案编译器）**

* **功能**：可视化的`.brew`方案创作工具。用户可拖拽参数曲线，即时获得AI模拟的风味预测。
* **定位**：面向进阶玩家和竞赛选手。Freemium模式（基础功能免费，高级AI分析付费）。
* **技术实现**：Web应用，需要后端AI模型支持。

**C. BrewScope（个人仪表盘）**

* **功能**：个人的冲煮日志与统计分析。自动识别“你的深烘豆萃取率总是偏高”等问题。
* **定位**：付费会员功能。

**D. BrewRepo（公共参数库）**

* **功能**：`.brew`文件的公共仓库。支持Fork、Star、评论、Pull Request。
* **定位**：社区的核心交互层，对标GitHub。

**E. AI诊断服务**

* **功能**：用户输入问题（如“这杯偏涩，后段发苦”），AI读取其历史`.brew`数据和当前设备信息，给出具体到参数级别的调整建议。
* **定位**：核心盈利引擎。单次付费或月费订阅。

### 3.3 技术服务层

* **Flavor LLM**：用全球`.brew`数据微调的垂直领域AI模型。核心能力是将模糊的风味描述翻译为精确的参数调整指令。
* **风味基因组数据库**：所有用户参数与评分数据构成的高维向量空间。每支豆子和每位用户在其中都有精确坐标。用于推荐和匹配。
* **`.brew`解析器API**：提供给智能咖啡设备厂商的SDK，使其硬件能直接读取并执行`.brew`文件。

## 第五部分：补充与调整建议

### 5.1 创始人人设与语气指南

**你是谁：**&#x4E00;个用自然语言指挥AI建站的“非典型开发者”。不装懂，不粉饰，不带官方腔。说话像跟朋友在深夜聊一个激动的想法。

**书写与表达原则：**

* **用“我”而不是“我们”**——除非确实在指代整个社区。
* **用主动句而非被动句**——“我写了这段代码”，而不是“这段代码被写了出来”。
* **允许粗糙与不完美**——这是开源项目的气质，不是商业发布。
* **用具体的数字和细节代替模糊的大词**——“50个种子文件”比“大量基础数据”有力。
* **允许幽默和自嘲**——“我一个村儿里的，给全世界咖啡极客定标准，这事儿本身就挺朋克的。”

### 5.2 进一步挖掘的资源与方向

**资源1：咖啡感官词库的标准化**

* 目前风味描述极度混乱。可发起一个“BrewCode Flavor Wheel”项目，用`.brew`文件的数据反向定义风味标签。

**资源2：与农业科技的结合**

**资源3：硬件商合作备忘录**
